[settings registration kill switch]
Goal: remove the old settings registration pattern entirely, not rename or polish it.

Definition of the forbidden pattern:
- arrays/registries of field metadata that mix API schema + grouping + storage mapping in one object
- any shared registration seam for settings fields
- any app-owned template file whose job is to register settings fields into a core package
- any test-support import whose only purpose is to trigger that registration side effect

Replacement rule:
- shared files own real schemas only
- optional small grouping metadata may exist separately when needed
- storage mapping lives in server resource/repository code only
- generated app templates do not install settings-field registration files
