# ESLint Config Package Plan (`@jskit/config-eslint`)

Status: Proposed  
Date: 2026-02-23  
Difficulty: 2

## 1. Objective

Create a shared ESLint config package so linting behavior stays consistent across SaaS apps and shared packages.

## 2. Non-Negotiable Boundaries

1. No app-specific globals.
2. No app-specific ignore patterns.
3. No app-specific architecture rules hardcoded in global config.
4. No auto-fixing assumptions that only fit one app.

## 3. Scope

## In scope

1. Base JS/TS lint policy.
2. Node and browser variants.
3. Vue (or framework) variant if broadly reused.
4. Import boundary lint hooks as optional layer.

## Out of scope

1. Per-app folder boundaries.
2. Per-app rule exceptions.
3. Formatting decisions handled by Prettier.

## 4. Package Structure

```text
packages/tooling/config-eslint/
  package.json
  README.md
  base.js
  node.js
  web.js
  vue.js
```

## 5. Public API

Flat config exports:

```js
import { baseConfig, nodeConfig, webConfig, vueConfig } from "@jskit/config-eslint";

export default [
  ...baseConfig,
  ...webConfig,
  ...vueConfig
];
```

## 6. Design Rules

1. Keep defaults strict but practical.
2. Keep rule overrides local to app when needed.
3. Keep boundary rules opt-in and parameterized.

## 7. Migration Plan

## Phase 1

1. Package scaffold and baseline rules.
2. Apply to one server package and one web package.

## Phase 2

1. Migrate all workspaces.
2. Remove duplicated root/app lint configs.

## Phase 3

1. Add optional reusable boundary helper config.

## 8. Test Plan

1. Lint fixture for Node module.
2. Lint fixture for web module.
3. Lint fixture for Vue if included.
4. Rule regression tests for key guardrails.

## 9. Acceptance Criteria

1. All app/package lint configs import from shared package.
2. No app-specific rules in package.
3. CI lint behavior consistent across workspaces.

## 10. Versioning Rules

1. Patch: docs/non-breaking rule tweaks.
2. Minor: additive rules disabled by default or warnings.
3. Major: new errors that can break CI broadly.

## 11. Anti-Abstraction Guardrails

1. No giant "do everything" config export.
2. Keep variants focused (`node`, `web`, `vue`).
3. Keep unsafe rule decisions local.

## 12. Work Breakdown

1. Build config package.
2. Add variants.
3. Add fixtures.
4. Migrate consumers.
5. Enforce in CI.

## 13. Definition of Done

1. `@jskit/config-eslint` adopted by apps and shared packages.
2. Duplicate eslint config removed.
3. CI uses only shared lint package.
