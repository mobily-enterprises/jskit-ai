# Entitlements Core Package Plan (`@jskit/entitlements-core`)

Status: Proposed  
Date: 2026-02-23  
Difficulty: 11

## 1. Objective

Extract a reusable entitlement domain engine that supports grants, consumption, balances, and limit resolution across SaaS apps, while keeping storage and product policy app-owned.

This is high value and high risk. Scope discipline is critical.

## 2. Non-Negotiable Boundaries

1. No SQL/ORM calls in core package.
2. No app-specific entitlement codes.
3. No app-specific product-plan mapping.
4. No HTTP/controller concerns.

## 3. Scope

## In scope

1. Domain entities and invariants.
2. Idempotent consumption orchestration.
3. Balance recomputation semantics.
4. Effective limitation resolution algorithm.
5. Repository interface contracts.

## Out of scope

1. Table schemas.
2. Migrations.
3. Provider checkout/subscription workflows.
4. Product catalog policy.

## 5. Public API

```js
export function createEntitlementsService(deps, options);
```

Dependencies:

1. `repository` (required)
2. `clock` (optional)
3. `logger` (optional)
4. `policy` (optional customization)

Core operations:

1. `grant(input)`
2. `consume(input)`
3. `recompute(input)`
4. `resolveEffectiveLimitations(subject)`

## 6. Domain Rules

1. Consumption operations must be idempotent by key.
2. Balance updates must be deterministic.
3. Recompute must be safe to rerun.
4. Operations support explicit transaction context from adapter.

## 7. Migration Plan

## Phase 1

1. Extract pure domain entities and error codes.
2. Add heavy unit tests with fixtures.

## Phase 2

1. Extract service orchestration with repository interface.
2. Keep existing repository implementation local.

## Phase 3

1. Replace app billing service entitlement section with core service.
2. Add behavior parity tests.

## Phase 4

1. Harden performance with profiling on critical paths.
2. Only then consider shared DB adapter.

## 8. Test Plan

1. Idempotency under repeated consume calls.
2. Race/concurrency simulation for consume+recompute.
3. Time-window grant behavior.
4. Negative/overflow amount handling.
5. Deterministic effective limitations output.

## 9. Acceptance Criteria

1. Core package has no schema or product coupling.
2. Existing entitlement behavior preserved.
3. At least two distinct app policy profiles can use core unchanged.

## 10. Versioning Rules

1. Patch: bugfix.
2. Minor: additive operations/options.
3. Major: semantic changes in entitlement computation.

## 11. Anti-Abstraction Guardrails

1. Do not add generic query builders.
2. Keep repository interface narrow and use-case-driven.
3. Prefer explicit operations over overly generic methods.

## 12. Work Breakdown

1. Extract entities/errors.
2. Define repository contract.
3. Port orchestration service.
4. Add tests.
5. Integrate with current app.

## 13. Definition of Done

1. Entitlement semantics live in shared core.
2. Storage remains adapter-injected.
3. Compatibility/perf tests pass.
