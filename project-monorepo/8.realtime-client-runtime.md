# Realtime Client Runtime Package Plan (`@jskit/realtime-client-runtime`)

Status: Proposed  
Date: 2026-02-23  
Difficulty: 8

## 1. Objective

Extract reusable client-side realtime runtime mechanics (connect/reconnect/subscribe/ack/defer) while keeping app-specific event handling local.

## 2. Non-Negotiable Boundaries

1. No app-specific topic names or event names in package.
2. No query cache invalidation logic in package.
3. No workspace/auth store type assumptions in package.
4. No UI framework coupling.

## 3. Scope

## In scope

1. Connection lifecycle and reconnect policy.
2. Subscription control messages.
3. Request tracking and ack/error correlation.
4. Deferred event replay hooks.
5. Transport abstraction interface.

## Out of scope

1. Event-to-cache invalidation mapping.
2. Domain-specific event routing.
3. Store-specific permission checks.

## 4. Package Structure

```text
packages/realtime/realtime-client-runtime/
  package.json
  README.md
  src/
    index.js
    runtime.js
    commandTracker.js
    transportContract.js
    policies/
      reconnect.js
      replay.js
  test/
    runtime.test.js
    commandTracker.test.js
```

## 5. Public API

```js
export function createRealtimeRuntime(options);
export function createCommandTracker(options);
```

Options include injection points:

1. `transport` (socket abstraction)
2. `resolveEligibility(context)`
3. `onEvent(event, context)`
4. `onConnectionStateChange(state)`
5. `isEventDeferred(event, context)`

## 6. Migration Plan

## Phase 1

1. Extract command tracker first (lowest-risk subset).
2. Keep runtime logic in app during initial extraction.

## Phase 2

1. Extract runtime state machine with injection hooks.
2. Use app adapters for auth/workspace/topic eligibility.

## Phase 3

1. Migrate all app runtime callers.
2. Remove legacy runtime implementation.

## 7. Test Plan

1. Connect/disconnect lifecycle.
2. Reconnect backoff boundaries.
3. Subscribe ack/error flow.
4. Deferred replay limits.
5. Duplicate command id handling.
6. Terminal-forbidden handling behavior.

## 8. Acceptance Criteria

1. Runtime package has zero app-domain nouns.
2. Existing realtime runtime tests pass through adapters.
3. All event handling specifics stay in app layer.

## 9. Versioning Rules

1. Patch: state-machine bugfixes.
2. Minor: additive callbacks/options.
3. Major: lifecycle contract changes.

## 10. Anti-Abstraction Guardrails

1. No implicit global stores.
2. No internal plugin framework.
3. Keep runtime deterministic and inspectable.

## 11. Work Breakdown

1. Extract command tracker.
2. Extract runtime core.
3. Add adapter interfaces.
4. Add tests.
5. Integrate and remove old code.

## 12. Definition of Done

1. Client realtime runtime is package-backed.
2. App-specific handlers remain app-local.
3. Realtime integration tests pass unchanged.
