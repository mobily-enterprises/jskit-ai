# RBAC Core Package Plan (`@jskit/rbac-core`)

Status: Proposed  
Date: 2026-02-23  
Difficulty: 5

## 1. Objective

Extract pure role/permission evaluation logic into a shared package usable by every SaaS app.

Goal: one consistent permission engine, zero transport/DB coupling.

## 2. Non-Negotiable Boundaries

1. No database queries.
2. No framework dependencies (`fastify`, `vue`, etc.).
3. No app-specific permission strings hardcoded as behavior.
4. No request/session assumptions.

## 3. Scope

## In scope

1. Manifest normalization and validation.
2. Role inheritance expansion.
3. Effective permission resolution.
4. Permission checking helpers.
5. Optional wildcard policy support.

## Out of scope

1. Membership lookup.
2. Tenant scoping and data access.
3. Route guard integration details.

## 4. Package Structure

```text
packages/auth/rbac-core/
  package.json
  README.md
  src/
    index.js
    manifest.js
    engine.js
    validators.js
  test/
    manifest.test.js
    engine.test.js
```

## 5. Public API

```js
export function validateManifest(manifest, options);
export function normalizeManifest(manifest, options);
export function createRbacEngine(manifest, options);

// engine
engine.has(permission, subject);
engine.hasAny(permissions, subject);
engine.hasAll(permissions, subject);
engine.listEffectivePermissions(subject);
```

Subject input stays generic:

```js
{
  roles: ["owner", "member"],
  grants: ["feature.x"],
  denies: ["feature.y"]
}
```

## 6. Migration Plan

## Phase 1

1. Port existing pure RBAC logic.
2. Add fixture-based tests for current behavior.

## Phase 2

1. Replace duplicated `hasPermission` implementations with shared engine calls.
2. Keep existing manifests local to each app.

## Phase 3

1. Remove old helper duplication.
2. Add architecture test to prevent re-duplication.

## 7. Test Plan

1. Role inheritance cycles rejected.
2. Unknown role handling.
3. Wildcard permission behavior.
4. Grants/denies precedence.
5. Deterministic output ordering.

## 8. Acceptance Criteria

1. Shared engine used by server and client permission checks where applicable.
2. No app-specific permission behavior in package code.
3. Existing permission outcomes preserved under migration tests.

## 9. Versioning Rules

1. Patch: bugfixes.
2. Minor: additive API helpers.
3. Major: semantic permission evaluation changes.

## 10. Anti-Abstraction Guardrails

1. No policy DSL.
2. No plugin system.
3. Keep algorithm explicit and testable.

## 11. Work Breakdown

1. Scaffold package.
2. Port and test manifest normalization.
3. Port and test engine evaluation.
4. Migrate call sites.
5. Remove duplicates.

## 12. Definition of Done

1. `@jskit/rbac-core` is the only RBAC evaluator.
2. Duplicate permission helpers are removed.
3. All permission tests pass unchanged.
