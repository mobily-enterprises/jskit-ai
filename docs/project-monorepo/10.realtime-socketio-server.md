# Realtime Socket.IO Server Package Plan (`@jskit/realtime-socketio-server`)

Status: Proposed  
Date: 2026-02-23  
Difficulty: 10

## 1. Objective

Extract reusable Socket.IO realtime server lifecycle mechanics while keeping authorization, topic policy, and event vocabulary app-local.

## 2. Non-Negotiable Boundaries

1. No hardcoded topic names.
2. No hardcoded surface IDs.
3. No hardcoded permission strings.
4. No app-specific workspace/user schema assumptions.
5. No domain event production in package.

## 3. Scope

## In scope

1. Socket connection lifecycle.
2. Protocol message parsing and validation.
3. Subscribe/unsubscribe orchestration.
4. Room naming strategy hooks.
5. Fanout helpers.
6. Optional Redis adapter wiring hooks.

## Out of scope

1. Actor identity source.
2. Membership and permission resolution.
3. Topic policy definitions.
4. Domain event publication semantics.

## 4. Package Structure

```text
packages/realtime/realtime-socketio-server/
  package.json
  README.md
  src/
    index.js
    registerServer.js
    protocol.js
    rooms.js
    adapters/
      redisStreams.js
    contracts.js
  test/
    protocol.test.js
    subscriptions.test.js
    rooms.test.js
```

## 5. Public API

```js
export function registerRealtimeSocketServer(io, deps, options);
```

Required deps:

1. `resolveConnectionContext(socket)`
2. `authorizeConnection(context)`
3. `authorizeTopic(context, topic)`
4. `resolveRoomKey(context, topic)`
5. `onSubscribe(context, topics)` (optional)
6. `onUnsubscribe(context, topics)` (optional)

## 6. Migration Plan

## Phase 1

1. Extract protocol helpers and room helpers first.
2. Keep main register function local.

## Phase 2

1. Extract register lifecycle into package.
2. Inject all app-specific policies via deps.

## Phase 3

1. Add Redis adapter wrapper as optional module.
2. Migrate server bootstrap to package entrypoint.

## 7. Test Plan

1. Message size limits and invalid payload paths.
2. Unauthorized/forbidden handling.
3. Subscribe/unsubscribe idempotency.
4. Room naming determinism.
5. Graceful shutdown behavior.
6. Redis adapter lifecycle behavior.

## 8. Acceptance Criteria

1. Package contains no domain vocabulary.
2. Server behavior remains parity under compatibility tests.
3. App-specific auth/authorization fully injected.

## 9. Versioning Rules

1. Patch: bugfixes.
2. Minor: additive hooks/options.
3. Major: lifecycle contract changes.

## 10. Anti-Abstraction Guardrails

1. No hidden state outside explicit context.
2. No internal policy engines.
3. Keep adapter contracts explicit.

## 11. Work Breakdown

1. Extract protocol + room utilities.
2. Extract server registration.
3. Add hooks/contracts.
4. Add tests.
5. Migrate app integration.

## 12. Definition of Done

1. Socket lifecycle core is package-owned.
2. App policies stay local.
3. Realtime server tests pass unchanged.
