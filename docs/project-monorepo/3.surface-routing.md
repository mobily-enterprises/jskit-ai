# Surface Routing Package Plan (`@jskit-ai/surface-routing`)

Status: Proposed  
Date: 2026-02-23  
Owner: Platform
Difficulty: 3

## 1. Goal

Extract reusable surface routing logic into a package that:

1. Works for the current app immediately.
2. Supports different surface lists per app (not hardcoded `app/admin/console`).
3. Keeps behavior parity while reducing boilerplate.
4. Enables shared fixes across apps through package upgrades.

## 2. Why This Package First

This is the lowest-risk, highest-confidence first extraction because:

1. Logic is pure and deterministic.
2. No DB coupling.
3. Already used on both client and server.
4. Existing tests already validate behavior.

## 3. Scope

## In scope

1. Current logic from:
   - `shared/routing/surfaceRegistry.js`
   - `shared/routing/surfacePaths.js`
2. Make registry configurable per app.
3. Provide safe defaults for backwards compatibility.
4. Add package-level tests and docs.

## Out of scope

1. App-specific route trees.
2. Auth policy decisions.
3. Workspace membership rules.
4. Rewriting router architecture.

## 5. API Design

Design rule: package owns logic; each app owns its surface definition.

## 5.1 Types (conceptual)

```ts
type SurfaceId = string;

type SurfaceDefinition = {
  id: SurfaceId;
  prefix: string;
  requiresWorkspace: boolean;
};

type SurfaceRegistry = Record<SurfaceId, SurfaceDefinition>;
```

## 5.2 Public API

```js
// defaults
export const DEFAULT_SURFACES;
export const DEFAULT_SURFACE_ID;

// registry helpers
export function defineSurfaceRegistry(input, options);
export function listSurfaceDefinitions(registry);
export function normalizeSurfaceId(surfaceId, registry, defaultSurfaceId);
export function resolveSurfacePrefix(surfaceId, registry, defaultSurfaceId);
export function surfaceRequiresWorkspace(surfaceId, registry, defaultSurfaceId);

// path helpers
export function normalizePathname(pathname);
export function matchesPathPrefix(pathname, prefix);
export function resolveSurfaceFromApiPathname(pathname, registry, defaultSurfaceId);
export function resolveSurfaceFromPathname(pathname, registry, defaultSurfaceId);
export function withSurfacePrefix(surfaceId, path, registry, defaultSurfaceId);
export function createSurfacePaths(surfaceId, registry, defaultSurfaceId);
export function resolveSurfacePaths(pathname, registry, defaultSurfaceId);
```

## 5.3 Backwards Compatibility

`DEFAULT_SURFACES` will mirror current behavior:

1. `app` prefix `""`, `requiresWorkspace: true`
2. `admin` prefix `"/admin"`, `requiresWorkspace: true`
3. `console` prefix `"/console"`, `requiresWorkspace: false`

Apps that do not pass a registry get existing behavior unchanged.

## 6. Configuration Model Per App

Each app defines local surfaces and passes them to shared logic.

Example app config:

```js
import { defineSurfaceRegistry } from "@jskit-ai/surface-routing";

export const APP_SURFACES = defineSurfaceRegistry(
  {
    app: { prefix: "", requiresWorkspace: true },
    admin: { prefix: "/admin", requiresWorkspace: true },
    partner: { prefix: "/partner", requiresWorkspace: false }
  },
  { defaultSurfaceId: "app" }
);
```

Then use:

```js
import { createSurfacePaths, resolveSurfaceFromPathname } from "@jskit-ai/surface-routing";
import { APP_SURFACES } from "./surfaceConfig.js";

const surfaceId = resolveSurfaceFromPathname(pathname, APP_SURFACES.registry, APP_SURFACES.defaultSurfaceId);
const paths = createSurfacePaths(surfaceId, APP_SURFACES.registry, APP_SURFACES.defaultSurfaceId);
```

This addresses the exact concern that each app will have a different surface list.

## 7. Migration Plan

## Phase 0: Baseline

1. Snapshot current tests:
   - `tests/surfacePathsAndRegistry.test.js`
   - client router/shell tests touching surface paths
   - server tests touching `resolveSurfaceFromPathname`
2. Record current behavior for:
   - API path resolution
   - browser path resolution
   - workspace slug extraction

## Phase 1: Create Package

1. Create package files and exports.
2. Copy current logic first with no behavior changes.
3. Add default registry module for backward compatibility.

Deliverable: package builds and tests pass in isolation.

## Phase 2: Add Configurable Registry

1. Refactor logic signatures to accept:
   - `registry`
   - `defaultSurfaceId`
2. Keep optional params so legacy calls still work.
3. Ensure normalization and prefix matching still match current behavior.

Deliverable: same behavior under default config; new behavior possible under custom config.

## Phase 3: Integrate Without Breaking Imports

1. Replace existing `shared/routing/*` files with thin re-export shims:
   - `shared/routing/surfaceRegistry.js`
   - `shared/routing/surfacePaths.js`
2. Keep all existing app imports working.
3. Run full tests.

Deliverable: zero runtime change, package now source of truth.

## Phase 4: Migrate Imports

1. Update imports in client/server from `shared/routing/*` to package path.
2. Keep shims for one release cycle.
3. Remove shims once migration complete.

Deliverable: no code depends on old shared path.

## Phase 5: Multi-App Ready

1. For each app, add `surfaceConfig` local module.
2. Pass app-local registry into package helpers.
3. Validate app-specific surfaces with tests.

Deliverable: one package, multiple surface profiles.

## 9. Acceptance Criteria

Extraction is complete when all are true:

1. Package exists and is used as source of truth.
2. Existing behavior unchanged under defaults.
3. Custom surface registry works in tests.
4. No app logic imports from another app.
5. CI test suite passes.
6. README documents public API and non-goals.

## 10. Risks And Mitigations

## Risk 1: Hidden coupling to hardcoded constants

Mitigation:

1. Keep defaults.
2. Add explicit custom-registry tests.
3. Use gradual migration with shims.

## Risk 2: Breaking path behavior during refactor

Mitigation:

1. Copy-as-is first.
2. Refactor signatures second.
3. Lock behavior with compatibility tests.

## Risk 3: Over-abstracting too early

Mitigation:

1. Only extract logic already shared.
2. Keep app policy/config local.
3. Avoid adding unrelated utilities to this package.

## 11. Release Plan

1. Version `0.1.0`: default-compatible extraction with shims.
2. Version `0.2.0`: custom registry API used by one app.
3. Version `1.0.0`: stable API after at least two apps adopt.

Semver rules:

1. `patch`: bugfix with no API break.
2. `minor`: additive API.
3. `major`: behavior or API breaking changes.
