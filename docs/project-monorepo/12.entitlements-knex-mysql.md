# Entitlements Knex MySQL Adapter Plan (`@jskit/entitlements-knex-mysql`)

Status: Proposed  
Date: 2026-02-23  
Difficulty: 12 (hardest)

## 1. Objective

Provide an optional Knex/MySQL adapter for `@jskit/entitlements-core` that is reusable only where schema and performance constraints are compatible.

This package should be created last.

## 2. Non-Negotiable Boundaries

1. No product-specific entitlement codes in SQL.
2. No app-specific joins.
3. No hidden assumptions about tenant model.
4. No migration ownership over app databases unless explicitly opted in.

## 3. Scope

## In scope

1. Repository contract implementation for core service.
2. Transaction-aware methods.
3. Optional migration helpers for adapter tables.
4. Query performance indexing guidance.

## Out of scope

1. Full app schema management.
2. Billing provider logic.
3. Product plan policy.

## 4. Package Structure

```text
packages/billing/entitlements-knex-mysql/
  package.json
  README.md
  src/
    index.js
    repository.js
    transactions.js
    migrations.js
    sql/
      schema.sql
      indexes.sql
  test/
    repository.integration.test.js
    transaction.test.js
```

## 5. Public API

```js
export function createEntitlementsKnexRepository(options);
export function withTransaction(knex, fn);
export function createEntitlementMigrations(options);
```

Options include:

1. `knex`
2. `tableNames` override
3. `dialectFeatures` flags
4. `clock` injection for deterministic tests

## 6. Design Rules

1. Keep methods aligned exactly with `entitlements-core` repository contract.
2. Avoid generic repository methods not required by core.
3. Keep SQL explicit and inspectable.

## 7. Migration Plan

## Phase 1

1. Implement adapter for one app schema profile.
2. Run integration tests against real MySQL.

## Phase 2

1. Prove compatibility with second app.
2. Add table-name and key-shape parameterization only where necessary.

## Phase 3

1. Publish adapter after two successful adopters.
2. Document strict compatibility matrix.

## 8. Test Plan

1. Integration tests for every repository method.
2. Transaction rollback/commit behavior.
3. Idempotency under concurrent inserts.
4. Recompute query performance on realistic data volume.
5. Migration up/down safety tests.

## 9. Acceptance Criteria

1. Adapter passes all `entitlements-core` contract tests.
2. No app-specific SQL remains.
3. p95 latency is acceptable on representative datasets.
4. Two SaaS apps use adapter without schema hacks.

## 10. Versioning Rules

1. Patch: SQL bugfixes without contract change.
2. Minor: additive options and indices.
3. Major: schema or contract changes requiring migration.

## 11. Anti-Abstraction Guardrails

1. Do not hide SQL behind a generic DSL.
2. Do not support every dialect; be explicit about MySQL scope.
3. If apps diverge too far, prefer separate adapters over one bloated one.

## 12. Work Breakdown

1. Implement repository contract methods.
2. Add transactions wrapper.
3. Add migration helpers.
4. Add integration/performance tests.
5. Pilot in two apps.

## 13. Definition of Done

1. Adapter is stable and contract-compliant.
2. Adapter remains domain-neutral.
3. Proven multi-app compatibility without custom forks.
