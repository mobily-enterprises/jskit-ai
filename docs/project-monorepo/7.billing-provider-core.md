# Billing Provider Core Package Plan (`@jskit/billing-provider-core`)

Status: Proposed  
Date: 2026-02-23  
Difficulty: 7

## 1. Objective

Extract provider-facing billing contracts so SaaS apps can plug in payment providers consistently without sharing product-specific billing logic.

## 2. Non-Negotiable Boundaries

1. No pricing catalogs in package.
2. No entitlement mapping logic in package.
3. No DB/repository code in package.
4. No provider-specific business policy baked into core.

## 3. Scope

## In scope

1. Provider adapter interface.
2. Webhook translation interface.
3. Standardized provider error taxonomy.
4. Provider registry helper.

## Out of scope

1. Plan/product definitions.
2. Checkout orchestration policy.
3. Subscription lifecycle state storage.

## 4. Package Structure

```text
packages/billing/billing-provider-core/
  package.json
  README.md
  src/
    index.js
    contracts/
      providerAdapter.js
      webhookTranslator.js
      providerError.js
    registry.js
    validation.js
  test/
    registry.test.js
    error.test.js
```

## 5. Public API

```js
export class BillingProviderError extends Error {}
export function createProviderRegistry(options);
export function validateProviderAdapter(adapter);
export function validateWebhookTranslator(translator);
```

Contract expectations:

1. provider adapters expose stable lifecycle methods.
2. webhook translators produce neutral event shape.
3. errors map to stable, provider-agnostic codes.

## 6. Migration Plan

## Phase 1

1. Extract existing shared provider contracts and errors.
2. Add adapter validation tests.

## Phase 2

1. Move provider registry to package.
2. Keep provider implementations app-local initially.

## Phase 3

1. Optionally move common provider utilities only if reused by at least two apps.

## 7. Test Plan

1. Registry rejects duplicate provider codes.
2. Contract validation failures are explicit.
3. Error normalization consistency across providers.
4. Translator output shape validation.

## 8. Acceptance Criteria

1. Provider contracts standardized across apps.
2. Package has zero product-specific billing policy.
3. Provider integration tests pass using shared interfaces.

## 9. Versioning Rules

1. Patch: bugfixes.
2. Minor: additive interface fields with defaults.
3. Major: contract signature changes.

## 10. Anti-Abstraction Guardrails

1. No "universal billing engine" in this package.
2. Keep this package contract-centric.
3. Keep orchestrators in app/domain layer.

## 11. Work Breakdown

1. Scaffold package.
2. Port contracts and errors.
3. Port registry helper.
4. Add tests.
5. Migrate imports.

## 12. Definition of Done

1. Shared contract package adopted by at least one provider path.
2. Provider code compiles against shared interfaces.
3. No app/product semantics leaked into package.
