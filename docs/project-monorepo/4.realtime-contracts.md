# Realtime Contracts Package Plan (`@jskit/realtime-contracts`)

Status: Proposed  
Date: 2026-02-23  
Owner: Platform
Difficulty: 4

## 1. Objective

Create a shared realtime contracts package that is:

1. Reusable across multiple SaaS apps.
2. Easy to maintain and hard to misuse.
3. Free of app-specific topics, event names, permission strings, and workspace assumptions.
4. Compatible with current code during migration.

This package is the second extraction target after `@jskit/surface-routing`.

## 2. Non-Negotiable Boundary Rules

The shared package must contain zero app-specific domain content.

## 2.1 Must not exist in package

1. Topic names like `workspace_meta`, `chat`, `projects`.
2. Event names like `workspace.project.created`.
3. Permission strings like `workspace.billing.manage`.
4. Surface IDs like `app/admin/console`.
5. Entity assumptions like `workspaceId`, `threadId`, `projectId`.

## 2.2 Can exist in package

1. Protocol message type constants.
2. Protocol error code constants (generic wording).
3. Catalog builder utilities.
4. Topic capability resolution helpers.
5. Generic envelope and validator helpers.

Rule of thumb: package provides mechanics, apps provide vocabulary.

## 3. Why This Is Still “Easy” (If Scoped Correctly)

It is easy if we extract the truly generic slice first:

1. `shared/realtime/protocolTypes.js` is already generic.
2. `shared/realtime/topicRegistry.js` logic is reusable once parameterized.
3. Existing tests heavily cover protocol behavior and topic gate behavior.

It becomes hard only if we also try to centralize app domain event catalogs right now.

## 4. Scope

## In scope

1. Shared protocol constants and validators.
2. Generic topic catalog engine.
3. Generic permission resolution for topic subscriptions.
4. Thin compatibility layer for current app during migration.

## Out of scope

1. Current app topic list as a shared default.
2. Current app event type constants as shared defaults.
3. Socket runtime implementation.
4. Server publish orchestration logic.

## 5.2 Public API (Generic Only)

```js
// protocol
export const REALTIME_MESSAGE_TYPES;
export const REALTIME_ERROR_CODES;
export function normalizeRequestId(value);
export function normalizeTopics(value);

// catalog
export function createTopicCatalog(definition, options);
export function listTopics(catalog);
export function getTopicRule(catalog, topic);
export function isSupportedTopic(catalog, topic);
export function isTopicAllowedForSurface(catalog, topic, surface);
export function listTopicsForSurface(catalog, surface);
export function resolveRequiredPermissions(catalog, topic, surface);
export function hasTopicPermission(catalog, topic, permissions, surface, options);

// envelope
export function createEventEnvelope(input);
export function createProtocolErrorPayload(input);
export function createAckPayload(input);
export function createPongPayload(input);

// validation
export function validateInboundMessage(message, options);
```

## 5.3 Important API Characteristics

1. All functions are stateless, pure where possible.
2. `catalog` is explicitly passed, never global.
3. No package-level singleton state.
4. Plain objects only, no class hierarchy needed.

Protocol wording rule:

1. Use transport/domain-neutral error codes in the package.
2. Provide temporary compatibility aliases only during migration windows.

## 6. App-Local Realtime Catalog Pattern

Each SaaS app defines its own realtime vocabulary locally and passes it to the package.

Example app-local file:

```text
apps/saas-a/contracts/realtimeCatalog.js
```

Example contents:

```js
import { createTopicCatalog } from "@jskit/realtime-contracts";

export const TOPICS = Object.freeze({
  PROJECTS: "projects",
  CHAT: "chat"
});

export const EVENT_TYPES = Object.freeze({
  PROJECT_CREATED: "project.created",
  CHAT_MESSAGE_CREATED: "chat.message.created"
});

export const REALTIME_CATALOG = createTopicCatalog({
  [TOPICS.PROJECTS]: {
    subscribeSurfaces: ["app", "admin"],
    requiredAnyPermission: ["projects.read"]
  },
  [TOPICS.CHAT]: {
    subscribeSurfaces: ["app"],
    requiredAnyPermission: ["chat.read"]
  }
});
```

This keeps package generic and app semantics app-owned.

## 7. Migration Strategy (No Big Bang)

## Phase 0: Baseline Lock

1. Run and snapshot current realtime tests.
2. Confirm behavior in:
   - client runtime subscription filtering
   - server subscribe authorization
   - protocol error payload shape

## Phase 1: Package Scaffold + Protocol Extraction

Move generic protocol constants first:

1. Port `shared/realtime/protocolTypes.js` into package `protocol.js`.
2. Add neutral error code `CONTEXT_REQUIRED`.
3. Provide compatibility alias from legacy `WORKSPACE_REQUIRED` to `CONTEXT_REQUIRED`.
4. Add package tests for constants and payload helpers.

Deliverable: protocol imports can switch with zero behavior change.

## Phase 2: Generic Topic Catalog Engine

Refactor `shared/realtime/topicRegistry.js` logic into `topicCatalog.js`:

1. Remove hardcoded topic registry constant from package.
2. Accept catalog definition as input.
3. Keep function semantics aligned with current behavior.

Deliverable: logic is reusable, data remains app-specific.

## Phase 3: App Catalog Extraction

In current app, create app-local catalog:

1. Move current topic names and rules from `shared/realtime/topicRegistry.js` into app-local config.
2. Move current event names from `shared/realtime/eventTypes.js` into app-local contracts module.
3. Export app-local constants from one place.

Deliverable: no app vocabulary inside package.

## Phase 4: Compatibility Shims

To avoid broad breakage during transition:

1. `shared/realtime/protocolTypes.js` re-exports from package.
2. `shared/realtime/topicRegistry.js` uses app-local catalog + package functions.
3. `shared/realtime/eventTypes.js` re-exports app-local constants.

Deliverable: existing imports continue working while internals migrate.

## Phase 5: Import Migration

1. Update server/client modules to import from:
   - `@jskit/realtime-contracts` for protocol + catalog mechanics
   - app-local contracts module for topics/events
2. Keep shims for one release cycle, then remove.

Deliverable: clear final boundary.

## 8. Current Code Mapping

## Moves to package (generic)

1. Most of `shared/realtime/protocolTypes.js`.
2. Utility logic from `shared/realtime/topicRegistry.js`:
   - normalization
   - rule lookup
   - per-surface filtering
   - permission checks

## Stays app-local (specific)

1. Topic constants currently in `shared/realtime/eventTypes.js`.
2. Event type constants currently in `shared/realtime/eventTypes.js`.
3. Topic permission/surface policy values currently hardcoded in `shared/realtime/topicRegistry.js`.
4. Domain-specific envelope fields in `server/domain/realtime/services/events.service.js`.

## 9. Integration Points To Update

Primary touch points:

1. `server/realtime/registerSocketIoRealtime.js`
2. `src/services/realtime/realtimeRuntime.js`
3. `server/domain/realtime/services/events.service.js`
4. `server/modules/*` and `src/views/*` that currently import `eventTypes`.

Migration rule:

1. Mechanics from package.
2. Vocabulary from app-local contracts.

## 11. Acceptance Criteria

The extraction is done when:

1. `@jskit/realtime-contracts` has no app-specific topic/event names.
2. All topic and event vocabulary comes from app-local contracts.
3. Existing realtime behavior is unchanged in compatibility tests.
4. Server and client both use shared package mechanics.
5. Package README documents strict non-goals and integration pattern.

## 12. Versioning Plan

1. `0.1.0`: protocol constants + payload helpers.
2. `0.2.0`: generic catalog utilities.
3. `0.3.0`: envelope/validator helpers.
4. `1.0.0`: after two SaaS apps use distinct catalogs without API churn.

Semver policy:

1. Patch: bugfixes with identical public behavior.
2. Minor: additive APIs only.
3. Major: breaking contracts only with migration notes.

## 13. Anti-Abstraction Guardrails

To prevent “stupid abstraction”:

1. No plugin framework.
2. No deep inheritance or strategy object explosion.
3. No hidden magic defaults except explicit backwards-compat adapters.
4. No coupling to transport implementation (`socket.io`) in this package.
5. No coupling to auth/store/query-client types.

If a new requirement forces package internals to know app domain nouns, do not add it to this package.

## 14. Concrete Task Breakdown

1. Create package skeleton and exports.
2. Port protocol constants and helpers.
3. Add protocol tests.
4. Implement generic catalog helpers.
5. Add catalog tests.
6. Create app-local contracts module for current app topics/events.
7. Add compatibility shims in `shared/realtime/*`.
8. Switch server/client imports gradually.
9. Run full realtime test matrix.
10. Remove shims after one stable cycle.

## 15. Definition of Done

1. Package is workspace-consumable and publishable.
2. Package contains only generic mechanics.
3. App-specific realtime vocabulary is fully local.
4. Existing tests pass with no behavior regressions.
5. Second SaaS app can define a different catalog without package changes.
