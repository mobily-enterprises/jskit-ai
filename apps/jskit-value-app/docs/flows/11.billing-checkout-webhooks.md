# Billing Checkout + Webhook Flow Tutorial

This tutorial covers two related flows:
- user-initiated checkout start
- provider webhook processing

## A) Checkout start flow

### 1) HTTP route + required idempotency header

```js
// packages/billing/billing-fastify-adapter/src/controller.js
function requireIdempotencyKey(request) {
  const idempotencyKey = normalizeIdempotencyKey(request?.headers?.["idempotency-key"]);
  if (!idempotencyKey) {
    throw new AppError(400, "Idempotency-Key header is required.", {
      code: "IDEMPOTENCY_KEY_REQUIRED"
    });
  }

  return idempotencyKey;
}

async function startCheckout(request, reply) {
  requireIdempotencyKey(request);
  const response = await executeAction(actionExecutor, {
    actionId: BILLING_ACTION_IDS.CHECKOUT_START,
    request,
    input: request.body || {}
  });

  reply.code(200).send(response);
}
```

### 2) Action contributor routes to billing service

```js
// packages/billing/billing-service-core/src/actions/workspaceBilling.contributor.js
{
  id: "workspace.billing.checkout.start",
  kind: "command",
  channels: ["api", "internal"],
  surfaces: ["admin"],
  permission: requireWorkspaceBillingManageOrSelf,
  idempotency: "required",
  async execute(input, context) {
    return billingService.startCheckout({
      request: resolveRequest(context, input),
      user: resolveUser(context, input),
      payload: normalizeObject(input),
      clientIdempotencyKey: resolveIdempotencyKey(context, input)
    });
  }
}
```

### 3) Service delegates to orchestrator

```js
// billing-service-core/src/service.js
async function startCheckout({ request, user, payload, clientIdempotencyKey, now = new Date() }) {
  return billingCheckoutOrchestrator.startCheckout({
    request,
    user,
    payload,
    clientIdempotencyKey,
    now
  });
}
```

### 4) Orchestrator claims/replays idempotency

```js
// checkoutOrchestrator.service.js
claim = await billingIdempotencyService.claimOrReplay(
  {
    action: BILLING_ACTIONS.CHECKOUT,
    billableEntityId: billableEntity.id,
    clientIdempotencyKey,
    requestFingerprintHash,
    normalizedRequestJson: normalizedRequest,
    provider: activeProvider,
    now
  },
  { trx }
);
```

Then it either:
- replays prior response
- resumes pending state
- or creates a new provider checkout session

## B) Webhook flow

### 1) Raw body capture for signature verification

```js
// server/fastify/billingWebhookRawBody.plugin.js
fastify.addHook("preParsing", async (request, _reply, payload) => {
  if (!BILLING_WEBHOOK_PATH_SET.has(safePathnameFromRequest(request))) {
    return payload;
  }

  ...
  request.rawBody = Buffer.concat(chunks);
  ...
});
```

### 2) Controller passes provider + signature + raw body

```js
// billing-fastify-adapter controller
await billingWebhookService.processProviderEvent({
  provider: "stripe",
  rawBody,
  signatureHeader: request.headers?.["stripe-signature"]
});
```

### 3) Webhook service validates + canonicalizes

```js
// billing-service-core/src/webhook.service.js
const providerEvent = await scopedProviderAdapter.verifyWebhookEvent({
  rawBody,
  signatureHeader: signature,
  endpointSecret
});
const canonicalEvent = scopedWebhookTranslator.toCanonicalEvent(providerEvent);

if (!scopedWebhookTranslator.supportsCanonicalEventType(eventType)) {
  return { ignored: true, eventId: String(canonicalEvent?.id || ""), eventType };
}

const processed = await processEventTransaction(canonicalEvent, { provider: normalizedProvider });
```

### 4) Transactional processing updates projections

Inside `processEventTransaction(...)`, webhook event rows are lock-created/updated and billing projection services are applied with retry-safe semantics.

## Quick debugging map

- 400 missing idempotency header: controller pre-check
- 400/500 webhook signature errors: webhook service verify step
- duplicate checkout behavior: idempotency claim/replay branch
- stale pending checkout: orchestrator recovery/expiry logic
