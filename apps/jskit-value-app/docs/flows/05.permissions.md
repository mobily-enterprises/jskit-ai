# Permissions Flow Tutorial

This tutorial explains how permissions work end-to-end in this codebase.

It is written for junior engineers and focuses on:

1. Where permissions come from
2. How they are assigned
3. How they are enforced at runtime
4. How to debug permission failures

## 1) Source of truth: RBAC manifest

Workspace-role permissions start from `shared/rbac.manifest.json`.

```json
{
  "version": 1,
  "defaultInviteRole": "member",
  "roles": {
    "owner": {
      "assignable": false,
      "permissions": ["*"]
    },
    "admin": {
      "assignable": true,
      "permissions": [
        "workspace.members.invite",
        "workspace.members.view",
        "workspace.members.manage",
        "workspace.invites.revoke",
        "workspace.settings.view",
        "workspace.settings.update",
        "workspace.roles.view"
      ]
    }
  }
}
```

What this means:

- `owner` is special and non-assignable.
- `*` means full access.
- Invite flows use `defaultInviteRole` if no explicit role is sent.

## 2) Manifest is normalized at boot

The app loads and validates the manifest at startup.

```js
// server.js
const RBAC_MANIFEST = await loadRbacManifest(APP_CONFIG.rbacManifestPath);
```

Inside `@jskit-ai/rbac-core`, normalization enforces invariants:

```js
function assertValidOwnerRole(role) {
  if (role.assignable !== false) {
    throw new Error("RBAC manifest roles.owner must be non-assignable.");
  }

  const permissions = toUniqueStringArray(role.permissions);
  if (!permissions.includes("*")) {
    throw new Error('RBAC manifest roles.owner.permissions must include "*".');
  }
}
```

Permission helpers:

```js
function resolveRolePermissions(manifest, roleId) {
  const normalizedRoleId = String(roleId || "").trim().toLowerCase();
  if (normalizedRoleId === OWNER_ROLE_ID) {
    return ["*"];
  }
  const role = manifest?.roles?.[normalizedRoleId];
  if (!role) return [];
  return toUniqueStringArray(role.permissions);
}

function hasPermission(permissionSet, permission) {
  const required = String(permission || "").trim();
  if (!required) return true;
  if (!Array.isArray(permissionSet)) return false;
  return permissionSet.includes("*") || permissionSet.includes(required);
}
```

## 3) Workspace context computes effective permissions

When request context is resolved for workspace surfaces, permissions are derived from membership role.

```js
// workspace.service.js
function resolvePermissions(roleId) {
  const normalizedRoleId = String(roleId || "").trim().toLowerCase();
  if (!normalizedRoleId) return [];
  if (normalizedRoleId === OWNER_ROLE_ID) return ["*"];
  return resolveRolePermissions(rbacManifest, normalizedRoleId);
}
```

Then context returns the effective list:

```js
return {
  workspace: ...,
  membership: selection.selected ? mapMembershipSummary(selection.selected.membership) : null,
  permissions: selection.selected ? selection.selected.access.permissions : [],
  workspaces: selection.workspaces,
  userSettings: ...
};
```

Important:

- No active membership means no workspace permissions.
- Surface/workspace access policy can further reduce allowed access.

## 4) Route-level checks (defense-in-depth)

Routes still declare policy metadata:

```js
// workspace-fastify-adapter/src/routes/admin.route.js
{
  path: "/api/workspace/members/:memberUserId/role",
  method: "PATCH",
  auth: "required",
  workspacePolicy: "required",
  workspaceSurface: "admin",
  permission: "workspace.members.manage",
  handler: controllers.workspace?.updateWorkspaceMemberRole
}
```

That metadata is enforced in Fastify `preHandler`:

```js
// fastify-auth-policy/plugin.js
if (meta.permission) {
  const allowed = await Promise.resolve(
    hasPermission({
      permission: meta.permission,
      permissions: request.permissions,
      request,
      actor: actorResolution.actor,
      context,
      meta
    })
  );
  if (!allowed) {
    throw createError(403, "Forbidden.");
  }
}
```

This is a pre-check, not the final authority.

## 5) Action-level checks (authoritative)

Each action declares its permission policy:

```js
// workspace.contributor.js
{
  id: "workspace.member.role.update",
  permission: ["workspace.members.manage"],
  ...
}
```

The action pipeline enforces it centrally:

```js
// action-runtime-core/pipeline.js
const permissionResolution = await authorizationEvaluator.evaluate({
  definition,
  context: normalizedContext,
  input: normalizedInput
});

if (!permissionResolution?.allowed) {
  throw Object.assign(new Error("Forbidden."), {
    status: 403,
    statusCode: 403,
    code: permissionResolution?.code || "ACTION_PERMISSION_DENIED"
  });
}
```

Why this matters:

- API, assistant tools, and internal callers all pass through this same check.
- This is the canonical place for business authorization.

## 6) How workspace roles are assigned

Role assignment is constrained by assignable roles from manifest:

```js
// admin.service.js
function normalizeRoleForAssignment(roleId) {
  const normalized = String(roleId || "").trim();
  if (!normalized) {
    throw new AppError(400, "Validation failed.", {
      details: {
        fieldErrors: {
          roleId: "Role is required."
        }
      }
    });
  }
  if (!assignableRoleIds.includes(normalized)) {
    throw new AppError(400, "Validation failed.", {
      details: {
        fieldErrors: {
          roleId: "Role is not assignable."
        }
      }
    });
  }
  return normalized;
}
```

Updating a member role:

```js
async function updateMemberRole(workspaceContext, payload) {
  const memberUserId = parsePositiveInteger(payload?.memberUserId);
  const roleId = normalizeRoleForAssignment(payload?.roleId);
  const existingMembership = await workspaceMembershipsRepository.findByWorkspaceIdAndUserId(...);
  if (!existingMembership || existingMembership.status !== "active") {
    throw new AppError(404, "Member not found.");
  }
  await workspaceMembershipsRepository.updateRoleByWorkspaceIdAndUserId(workspace.id, memberUserId, roleId);
  return listMembers(workspace);
}
```

## 7) How console permissions are assigned

Console uses its own role system and access service:

```js
// consoleAccess.service.js
const membership = await consoleMembershipsRepository.findByUserId(userId);
const activeMembership = membership && membership.status === "active" ? membership : null;
const permissions = activeMembership ? resolveRolePermissions(activeMembership.roleId) : [];
```

And permission guard:

```js
async function requirePermission(user, permission) {
  const context = await requireConsoleAccess(user);
  if (!hasPermission(context.permissions, permission)) {
    throw new AppError(403, "Forbidden.");
  }
  return context;
}
```

Role update path:

```js
// consoleMembers.service.js
await requirePermission(user, CONSOLE_MANAGEMENT_PERMISSIONS.MEMBERS_MANAGE);
await ensureRootMutationAllowed(user, memberUserId);
const roleId = normalizeRoleForAssignment(payload?.roleId);
await consoleMembershipsRepository.updateRoleByUserId(memberUserId, roleId);
```

## 8) Invite flows also assign permissions (indirectly)

Workspace invite stores target role:

```js
// admin.service.js
const roleId = normalizeRoleForAssignment(payload?.roleId || rbacManifest.defaultInviteRole);
createdInvite = await workspaceInvitesRepository.insert({
  workspaceId: workspace.id,
  email,
  roleId,
  tokenHash: hashInviteToken(inviteToken),
  ...
});
```

Accepting invite creates/activates membership with that role:

```js
const roleId = normalizeRoleForAssignment(invite.roleId || rbacManifest.defaultInviteRole);
await workspaceMembershipsRepository.ensureActiveByWorkspaceIdAndUserId(
  invite.workspaceId,
  userId,
  roleId,
  transactionOptions
);
```

Same pattern exists for console invites (`ensureActiveByUserId(userId, roleId, ...)`).

## 9) End-to-end example: `workspace.member.role.update`

Practical chain:

1. Route metadata says `permission: "workspace.members.manage"` (pre-check).
2. Auth policy plugin resolves `request.permissions` and checks route permission.
3. Controller executes action `workspace.member.role.update`.
4. Action pipeline checks action permission `["workspace.members.manage"]`.
5. Admin service validates `roleId` is assignable and updates membership row.
6. Next requests recalculate permissions from new role.

## 10) Why you may see `403 Forbidden`

Most common causes:

1. Role does not include the required permission string.
2. User has no active membership (workspace or console).
3. Surface mismatch (`app` vs `admin` vs `console`).
4. Route metadata and action permission are not aligned.
5. Action context is missing permissions (bad request context construction).

Fast debug checklist:

1. Confirm user membership role in DB.
2. Confirm resolved permission array in request/action context.
3. Confirm route permission metadata.
4. Confirm action `permission` declaration.
5. Confirm surface and workspace scope are expected.

## 11) Big-picture mental model

Use this simple model:

1. **Manifest** defines role -> permissions.
2. **Membership** picks one role for a user in a scope.
3. **Context resolver** turns membership into effective `permissions`.
4. **Route pre-check** protects transport edge.
5. **Action pipeline** is authoritative for business authorization.

If you follow this order while debugging, permission issues become much easier to reason about.
