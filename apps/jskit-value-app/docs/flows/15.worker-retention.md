# Retention Worker Flow Tutorial

This tutorial follows retention work from enqueue to sweep execution.

## 1) Queue payload normalization and enqueue

```js
// packages/operations/redis-ops-core/src/retentionQueue.js
function normalizeRetentionSweepPayload(payload = {}) {
  const explicitIdempotencyKey = normalizeIdempotencyKey(payload.idempotencyKey);
  const resolvedIdempotencyKey = explicitIdempotencyKey ||
    (normalizeLabel(payload.trigger, "manual") === "cron"
      ? buildCronIdempotencyKey({ now: payload.now || new Date(), dryRun: Boolean(payload.dryRun) })
      : "");

  return {
    dryRun: Boolean(payload.dryRun),
    trigger: normalizeLabel(payload.trigger, "manual"),
    requestedBy: normalizeRequestedBy(payload.requestedBy),
    idempotencyKey: resolvedIdempotencyKey
  };
}
```

If idempotency key exists, jobId becomes deterministic (`retention-<key>`).

## 2) Worker runtime starts BullMQ worker

`server/workers/runtime.js` delegates to core runtime:

```js
return createWorkerRuntimeCore({
  ...runtimeOptions,
  isLockHeldError: isRetentionLockHeldError,
  connectionFactory,
  createRetentionSweepProcessorImpl,
  ...
});
```

## 3) Processor acquires distributed lock

```js
// redis-ops-core/src/retentionProcessor.js
lockAcquired = await acquireDistributedLockImpl({
  connection: lockConnection,
  key: lockKey,
  token: lockToken,
  ttlMs: normalizedLockTtlMs
});

if (!lockAcquired) {
  throw new RetentionLockHeldError({ jobId, lockKey, ... });
}
```

Only one active sweep across workers should run at a time.

## 5) Retention-core orchestrator runs rule pack

```js
// packages/operations/retention-core/src/service.js
const orchestrator = createRetentionSweepOrchestrator({
  rules,
  retentionConfig: normalizedRetentionConfig,
  batchSize: normalizedRetentionConfig.batchSize,
  now,
  failFast: true
});

return {
  runSweep: orchestrator.runSweep
};
```

Each rule can be dry-run or execute mode.

## 6) CLI enqueue helper

`server/workers/enqueueRetentionSweepCli.js` parses:
- `--dry-run`
- `--trigger=<...>`
- `--requested-by=<...>`
- `--idempotency-key=<...>`

Useful for safe operational testing.

## 7) Operational behavior summary

- lock contention retries/requeues are handled in worker runtime
- lock extension heartbeat protects long sweeps
- failures can route to dead-letter queue for operator triage
