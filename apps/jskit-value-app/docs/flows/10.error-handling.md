# Error Handling Flow Tutorial

This tutorial explains how errors become HTTP responses.

## 1) Canonical application error class

```js
// packages/runtime/server-runtime-core/src/errors.js
export class AppError extends Error {
  constructor(status, message, options = {}) {
    super(message);
    this.name = "AppError";
    this.status = Number(status) || 500;
    this.statusCode = this.status;
    this.code = options.code || "APP_ERROR";
    this.details = options.details;
    this.headers = options.headers || {};
  }
}
```

Use `AppError` for expected API failures.

## 2) Central Fastify error handler

    reply.code(400).send({
      error: "Validation failed.",

      details: { fieldErrors }

## 3) Validation error shape

The app intentionally preserves field-level structure:
- top-level `fieldErrors`
- mirrored `details.fieldErrors`

That keeps frontend form handling consistent.

## 4) Action pipeline errors

Action runtime throws structured runtime errors for:
- unknown action IDs
- channel/surface forbidden
- validation failures
- permission denied
- required idempotency key missing

Controllers do not need custom mapping for these; global handler returns API-safe payload.

## 5) Operational side effects for 5xx

For server errors, the handler also:
- records DB-like errors into observability counters when possible
- captures server error telemetry
- logs full error context

## 6) What to avoid

- throwing raw provider/SDK errors directly to clients
- returning ad-hoc error shapes from controller code
- bypassing `AppError` for expected business failures
