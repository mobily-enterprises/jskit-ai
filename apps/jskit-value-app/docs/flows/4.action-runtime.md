# Action Runtime Flow Tutorial

This walkthrough focuses on the canonical action path.

Goal:
- show where actions are registered
- show how `execute()` flows through the pipeline
- show where policy/audit/observability hooks run

## 1) App composes contributors explicitly

```js
// server/runtime/actions/contributorManifest.js
function createActionContributors({ services, repositories, repositoryConfig, appConfig, rbacManifest } = {}) {
  const actionConfig = repositoryConfig?.actions || {};

  return Object.freeze([
    createAuthActionContributor({ authService: services?.authService }),
    createWorkspaceActionContributor({ ... }),
    createConsoleActionContributor({ ... }),
    createChatActionContributor({ chatService: services?.chatService }),
    createWorkspaceBillingActionContributor({ ... }),
    createSettingsActionContributor({ ... }),
    createProjectsActionContributor({ ... }),
    createDeg2radHistoryActionContributor({ ... }),
    createAssistantActionContributor({ ... }),
    createConsoleErrorsActionContributor({ ... }),
    createCommunicationsActionContributor({ ... })
  ]);
}
```

Why this matters:
- no import-time side effects
- clear ownership by domain
- removing one import removes that contributor from runtime

## 2) App creates registry with adapters

```js
// server/runtime/actions/createActionRegistry.js
function createRuntimeActionRegistry({ contributors, services, logger = console } = {}) {
  return createActionRegistry({
    contributors,
    permissionEvaluator: createPermissionEvaluator(),
    idempotencyAdapter: createActionIdempotencyAdapter({
      billingIdempotencyService: services?.billingIdempotencyService
    }),
    auditAdapter: createActionAuditAdapter({ auditService: services?.auditService, logger }),
    observabilityAdapter: createActionObservabilityAdapter({
      observabilityService: services?.observabilityService,
      logger
    }),
    logger
  });
}
```

Think of this as dependency wiring, not business logic.

## 3) Controllers call action executor

```js
// server/runtime/actions/index.js
async execute({ actionId, version = null, input = {}, context = {}, deps = {} } = {}) {
  const executionContext = buildExecutionContext(context);

  return actionRegistry.execute({
    actionId,
    version,
    input,
    context: executionContext,
    deps: {
      ...(defaultDeps || {}),
      ...(deps || {})
    }
  });
}
```

Important:
- all channels go through the same executor shape
- context is normalized before registry execution

## 4) Registry resolves definition + executes pipeline

```js
// packages/runtime/action-runtime-core/src/registry.js
async function execute({ actionId, version = null, input = {}, context = {}, deps = {} } = {}) {
  const definition = resolveActionDefinition(index, actionId, version);
  const execution = await executeActionPipeline({
    definition,
    input,
    context,
    deps,
    permissionEvaluator,
    idempotencyAdapter,
    auditAdapter,
    observabilityAdapter,
    logger
  });

  return execution.result;
}
```

Definition lookup rules:
- if `version` omitted: latest version for that action ID
- duplicate `id+version`: startup error
- unknown action: `ACTION_NOT_FOUND`

## 5) Pipeline order (the core)

```js
// packages/runtime/action-runtime-core/src/pipeline.js
ensureActionChannelAllowed(definition, normalizedContext);
ensureActionSurfaceAllowed(definition, normalizedContext);
ensureActionVisibilityAllowed(definition, normalizedContext);

const normalizedInput = await normalizeActionInput(definition, input, normalizedContext);
const permissionResolution = await authorizationEvaluator.evaluate({ definition, context: normalizedContext, input: normalizedInput });

idempotencyKey = resolveActionIdempotencyKey(definition, normalizedContext);
ensureIdempotencyKeyIfRequired(definition, normalizedContext, idempotencyKey);

const executionResult = await definition.execute(normalizedInput, normalizedContext, deps);
const normalizedResult = await normalizeActionOutput(definition, executionResult, normalizedContext);
```

Then the pipeline:
- emits audit
- emits observability signals
- logs structured `action.execution`

## 6) Permission policy forms

```js
// packages/runtime/action-runtime-core/src/policies.js
if (typeof policy === "function") {
  const resolution = await policy(context, input);
  ...
}

const requiredPermissions = Array.isArray(policy) ? policy : [];
for (const permission of requiredPermissions) {
  if (!hasPermission(actorPermissions, permission)) {
    return { allowed: false, ... };
  }
}
```

You can declare:
- static list: `permission: ["workspace.members.manage"]`
- dynamic function: `permission: (context, input) => boolean | { allowed, code, reason }`

## 7) Mental model for juniors

When you add a new business capability:
1. add action definition in owning contributor
2. use domain service in `execute()`
3. define permission + idempotency + channels + surfaces
4. call that action from controller/tool/worker

Do not add a second business path in controller/service wrappers.
