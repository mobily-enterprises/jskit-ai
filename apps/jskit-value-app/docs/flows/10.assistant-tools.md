# Assistant Tools Flow Tutorial

This tutorial explains how assistant tools are derived from actions, scoped by surface, and executed.

## 1) App AI service injects dynamic tool resolver

```js
// server/modules/ai/services/ai.service.js
const resolveToolsForSurface = createAssistantActionToolsResolver({
  resolveActionExecutor,
  actionsConfig: source.actionsConfig
});

return createAssistantCoreService({
  ...source,
  resolveToolsForSurfaceFn: resolveToolsForSurface,
  tools: []
});
```

No hardcoded `workspace_rename` path anymore.

## 2) Resolver derives tools from action definitions

```js
// server/modules/ai/lib/tools/actionTools.js
const definitions = normalizeToolDefinitions(actionExecutor.listDefinitions(), {
  surfaceId: normalizedSurfaceId,
  actionsConfig
});
```

Filtering rules include:
- channel must include `assistant_tool`
- action surface list must include current surface
- `kind !== "stream"`
- visibility must not be `operator`
- app config allow/block lists

## 3) Tool name and idempotency are deterministic

```js
function createToolName(actionId, usedNames = new Map()) { ... }

function buildAssistantToolIdempotencyKey({ assistantMeta, actionId, args }) {
  ...
  return `assist:${conversationId}:${toolCallId}:${actionHash}:${actionArgsHash}`;
}
```

This keeps provider-safe names while preserving canonical action mapping.

## 4) Tool execution calls action executor

```js
return await actionExecutor.execute({
  actionId,
  version: actionVersion,
  input: normalizedArgs,
  context: {
    request: requestObject,
    actor: context?.user || requestObject?.user || null,
    workspace: context?.workspace || requestObject?.workspace || null,
    permissions: actorPermissions,
    channel: "assistant_tool",
    surface: normalizedSurfaceId,
    assistantMeta: { ...assistantMeta, toolName, actionId },
    requestMeta: { idempotencyKey }
  }
});
```

## 5) Assistant core scopes tools by surface per turn

```js
// packages/ai-agent/assistant-core/src/service.js
const {
  toolRegistry: scopedToolRegistry,
  providerTools: scopedProviderTools,
  allowedToolNames
} = await resolveToolsForSurface(assistantSurfaceId, request);
```

Prompt also states allowed tools explicitly:
- `Tools available in this turn: ...`

## 6) Tool call loop

```js
const toolResult = await executeToolCall(scopedToolRegistry, {
  name: toolName,
  args: parsedArgs,
  context: toolCallContext,
  appErrorClass: AppError,
  hasPermissionFn: permissionCheck
});
```

If a tool fails:
- error is mapped
- streamed as tool failure event
- audit and metrics are recorded for the turn

## 7) Surface isolation guarantee

- App surface does not receive non-app tools in provider schemas.
- Resolver filters definitions by `surfaces` before schema publication.
- Non-matching tools are omitted, not denied later at execution time.

## 8) Cross-tab contract

- Assistant **live stream events** are local-tab runtime only (no websocket mirror).
- Cross-tab consistency is transcript-sync based:
  - workspace transcript realtime topic invalidates transcript explorer queries
  - assistant workspace/root query families are invalidated from transcript events
- Surface policy allows transcript subscribe on:
  - `admin` with `workspace.ai.transcripts.read`
  - `app` when transcript-sync is enabled for end-user assistant UX
