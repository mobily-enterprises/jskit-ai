# Audit + Observability Flow Tutorial

This tutorial explains what gets emitted around action execution.

## 1) Action pipeline emits audit once

```js
// packages/runtime/action-runtime-core/src/pipeline.js
await emitAuditEvent(normalizedAuditAdapter, {
  definition,
  context: normalizedContext,
  outcome: "success",
  result: normalizedResult,
  error: null,
  durationMs,
  idempotencyReplay: false
});
```

Failure path also emits once with `outcome: "failure"`.

## 2) App audit adapter builds event payload

```js
// server/runtime/actions/auditAdapters.js
const auditEvent = {
  ...baseAuditPayload,
  action: normalizeText(definition?.audit?.actionName || definition?.id),
  outcome: normalizeOutcome(outcome),
  metadata: {
    ...metadata,
    actionId: normalizeText(definition?.id),
    actionVersion: Number(definition?.version || 1),
    channel: normalizeText(context?.channel),
    durationMs: Number(durationMs) || 0,
    idempotencyReplay: idempotencyReplay === true
  }
};

await auditService.recordSafe(auditEvent, request?.log);
```

## 3) Security audit service sanitizes metadata

```js
// packages/security/security-audit-core/src/service.js
const SENSITIVE_METADATA_KEY_PATTERN = /(pass(word)?|token|secret|authorization|cookie|api[-_]?key|credential)/i;
...
function sanitizeMetadataValue(value, { key = "", depth = 0 } = {}) {
  if (SENSITIVE_METADATA_KEY_PATTERN.test(key)) {
    return "[REDACTED]";
  }
  ...
}
```

This prevents secret leakage into audit storage.

## 4) Observability adapter emits runtime signals

```js
// server/runtime/actions/observabilityAdapters.js
safeRecord("recordActionExecution", {
  action: definition?.id,
  channel: context?.channel,
  surface: context?.surface,
  outcome,
  durationMs,
  errorCode,
  idempotencyReplay: idempotencyReplay === true
});
```

Also emits:
- authorization denied
- validation failures
- idempotent replay notices

## 5) Current implementation note

`observabilityAdapters.js` is wired and called by pipeline.
Whether metrics increment depends on methods available on `observabilityService` (`recordActionExecution*`, etc.).
So this is an explicit extension point.

## 6) Structured action log

Pipeline logs `action.execution` with:
- action ID/version
- request ID
- channel/surface
- duration
- outcome/error code
- replay flag

This is helpful for per-request forensic tracing.
