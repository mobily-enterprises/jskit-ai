# Idempotency Flow Tutorial

This system has three idempotency patterns:
- generic action-runtime policy hooks
- billing strong idempotency service
- chat domain-native idempotency with fingerprints + tombstones

## 1) Runtime policy gate

```js
// packages/runtime/action-runtime-core/src/idempotency.js
function resolveActionIdempotencyKey(definition, context) {
  const policy = normalizeText(definition?.idempotency || "none").toLowerCase();
  if (policy === "none") {
    return "";
  }

  return resolveRequestIdempotencyKey(context);
}

function ensureIdempotencyKeyIfRequired(definition, context, key) {
  const policy = normalizeText(definition?.idempotency || "none").toLowerCase();
  if (policy !== "required") {
    return;
  }
  ...throw ACTION_IDEMPOTENCY_KEY_REQUIRED...
}
```

## 2) Pipeline claim/replay hook

```js
// packages/runtime/action-runtime-core/src/pipeline.js
const claimResult = await executeIdempotencyClaim(normalizedIdempotencyAdapter, {
  definition,
  context: normalizedContext,
  input: normalizedInput,
  idempotencyKey
});

if (claimResult.type === "replay") {
  ...
  return { result: claimResult.replayResult, idempotencyReplay: true };
}
```

## 3) App adapter hook (currently conservative)

```js
// server/runtime/actions/idempotencyAdapters.js
if (!actionId.startsWith("workspace.billing.")) {
  return noopAdapter.claimOrReplay(payload);
}

return noopAdapter.claimOrReplay(payload);
```

Meaning:
- adapter exists as integration seam
- today it defaults to no-op behavior
- billing performs strong idempotency in domain services/orchestrators

## 4) Billing controller enforces header for mutators

```js
// billing-fastify-adapter controller
requireIdempotencyKey(request);
```

This fails fast before business execution.

## 5) Billing core claim/replay

In checkout/portal/payment-link flows, billing services call `billingIdempotencyService.claimOrReplay(...)`, manage lease fencing, and mark succeeded/failed.

## 6) Chat domain-native idempotency

```js
// chat-core service
const existing = await chatMessagesRepository.findByClientMessageId(...);
if (existing) {
  const matches = await doesMessageMatchIdempotencyFingerprint(existing, incomingFingerprint, scopedOptions);
  if (!matches) {
    throw createIdempotencyConflictError();
  }
  return { idempotencyStatus: "replayed", ... };
}

const tombstone = await chatIdempotencyTombstonesRepository.findByClientMessageId(...);
if (tombstone) {
  ...retry blocked or conflict...
}
```

For chat, `clientMessageId` + payload fingerprint are authoritative.

## 7) Quick decision guide

- payment mutation? use billing idempotency key + billing service claim/replay
- chat send/upload? use chat client IDs and domain-native idempotency logic
- simple command with safe retries? use optional/required action policy as needed
