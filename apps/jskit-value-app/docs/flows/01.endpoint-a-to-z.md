# Endpoint Evolution Tutorial: From `[]` to Full Feature

This tutorial shows an endpoint evolving in realistic stages:

1. a URL that always returns `[]`
2. add database access through a repository
3. add service business logic
4. add realtime fanout
5. add permission enforcement (route + action)

To keep this concrete, it maps to patterns already used in the `projects` flow.

---

## 0) Stage Zero: URL returns `[]`

Start with the smallest possible HTTP behavior.

```js
// initial controller shape (teaching starter)
async function list(request, reply) {
  reply.code(200).send([]);
}
```

And route wiring:

```js
{
  path: "/api/workspace/projects",
  method: "GET",
  handler: controllers.projects?.list
}
```

What this gives you:
- contract exists
- client can integrate quickly
- no data yet

External calls at this stage:
- `reply.code(200).send([])`: writes HTTP status/body to Fastify response, then socket flush is handled by Node runtime.

## 1) Add Repository (DB connection enters)

Now make data real by introducing repository functions that own SQL access.

Existing pattern:

```js
// server/modules/projects/repository.js
import { db } from "../../../db/knex.js";

async function repoListForWorkspace(workspaceId, page, pageSize, options = {}) {
  const client = resolveClient(options);
  const offset = (page - 1) * pageSize;

  const rows = await client("workspace_projects")
    .where({ workspace_id: workspaceId })
    .orderBy("created_at", "desc")
    .orderBy("id", "desc")
    .limit(pageSize)
    .offset(offset);

  return rows.map(mapProjectRowRequired);
}
```

External calls in this stage:
- `client("workspace_projects")...`: Knex query builder produces SQL and executes it through the configured MySQL connection.
- `await ...`: waits for DB response rows.
- `mapProjectRowRequired(row)`: converts DB snake_case columns into API/domain shape and normalizes types.

Important boundary:
- controller must not call Knex directly
- DB logic lives in repository only

---

## 2) Add Service (business logic enters)

Now move validation, pagination rules, and domain errors into a service.

Existing pattern:

```js
// server/modules/projects/service.js
async function list(workspaceContext, pagination) {
  const workspaceId = normalizeWorkspaceId(workspaceContext);
  const page = Math.max(1, Number(pagination?.page) || 1);
  const pageSize = Math.max(1, Math.min(100, Number(pagination?.pageSize) || 10));

  const total = await projectsRepository.countForWorkspace(workspaceId);
  const totalPages = Math.max(1, Math.ceil(total / pageSize));
  const safePage = Math.min(page, totalPages);
  const entries = await projectsRepository.listForWorkspace(workspaceId, safePage, pageSize);

  return {
    entries,
    page: safePage,
    pageSize,
    total,
    totalPages
  };
}
```

External calls in this stage:
- `projectsRepository.countForWorkspace(...)`: DB `COUNT(*)` to compute pagination.
- `projectsRepository.listForWorkspace(...)`: DB page fetch.
- `throw new AppError(...)`: emits structured domain/API error contract for the global error handler.

Why this stage matters:
- consistent validation rules
- reusable business logic across channels
- controller remains transport-only

## 3) Wire into Canonical Action Runtime (current architecture)

In this codebase, controller should execute actions, not services directly.

Controller pattern:

```js
// server/modules/projects/controller.js
async function list(request, reply) {
  const response = await executeAction(actionExecutor, {
    actionId: "projects.list",
    request,
    input: request.query || {}
  });
  reply.code(200).send(response);
}
```

Action definition pattern:

```js
// server/runtime/actions/contributors/projects.contributor.js
{
  id: "projects.list",
  version: 1,
  kind: "query",
  channels: ["api", "assistant_tool", "internal"],
  surfaces: ["admin"],
  permission: ["projects.read"],
  idempotency: "none",
  async execute(input, context) {
    const payload = normalizeObject(input);
    return resolvedProjectsService.list(resolveWorkspace(context, payload), {
      page: payload.page,
      pageSize: payload.pageSize
    });
  }
}
```

External calls in this stage:
- `actionExecutor.execute(...)`: normalizes context then calls action registry.
- action registry pipeline enforces channel/surface/permission/validation before handler execution.
- `resolvedProjectsService.list(...)`: service layer call (business logic).

## 4) Add Realtime (for mutating endpoints)

Reads usually do not publish events. Mutations do.

Action mutation example:

```js
// server/runtime/actions/contributors/projects.contributor.js
async function execute(input, context) {
  const workspace = resolveWorkspace(context, input);
  const response = await resolvedProjectsService.create(workspace, input);

  publishProjectRealtimeEvent({
    context,
    input,
    workspace,
    project: response?.project,
    operation: "created"
  });
  return response;
}
```

Publisher boundary:

```js
// server/realtime/publishers/projectPublisher.js
return publishSafely({
  publishMethod: publishProjectEvent,
  payload: {
    operation,
    workspace,
    project,
    ...buildPublishRequestMeta(request)
  },
  request,
  logCode,
  logContext
});
```

Correlation guardrail:

- Event-producing write routes must include `x-command-id` and `x-client-id`.
- Client transport runtime injects these headers for correlated write paths (projects/workspace admin/chat writes, invitation redeem, and related mutation routes).

Realtime event service:

```js
// packages/runtime/server-runtime-core/src/realtimeEventsService.js
function publishProjectEvent({ operation, workspace, project, commandId, sourceClientId, actorUserId }) {
  const envelope = createEventEnvelope({
    eventType,
    topic: resolvedRealtimeTopics.PROJECTS,
    workspace,
    entityType: "project",
    entityId: project?.id,
    commandId,
    sourceClientId,
    actorUserId,
    payload: {
      operation: normalizedOperation || "updated",
      projectId: Number(project?.id)
    }
  });

  realtimeEventsBus.publish(envelope);
  return envelope;
}
```

Socket fanout wiring:

```js
// packages/realtime/realtime-server-socketio/src/registerRealtimeServerSocketio.js
const unsubscribeEventsListener = realtimeEventsService.subscribe((eventEnvelope) => {
  void fanoutEvent(eventEnvelope).catch((error) => {
    appLogger.warn({ err: error }, "realtime.socketio.fanout_failed");
  });
});
```

External calls in this stage:
- `publishProjectEvent(...)`: creates normalized envelope and publishes to runtime bus.
- `realtimeEventsService.subscribe(...)`: socket runtime listens for envelopes.
- `io.in(roomName).fetchSockets()` + `socket.emit(...)`: Socket.IO finds room members and pushes event payloads.

## 5) Add Permissions (where stored, how enforced)

Permission data is stored in RBAC manifest:

```json
// shared/rbac.manifest.json
"roles": {
  "admin": {
    "permissions": [
      "projects.read",
      "projects.write"
    ]
  },
  "member": {
    "permissions": ["projects.read", "projects.write"]
  },
  "viewer": {
    "permissions": ["projects.read"]
  }
}
```

Loaded at boot:

```js
// server.js
const RBAC_MANIFEST = await loadRbacManifest(APP_CONFIG.rbacManifestPath);
```

Converted to effective permission set in workspace context:

```js
// workspace.service.js
function resolvePermissions(roleId) {
  const normalizedRoleId = String(roleId || "").trim().toLowerCase();
  if (!normalizedRoleId) return [];
  if (normalizedRoleId === OWNER_ROLE_ID) return ["*"];
  return resolveRolePermissions(rbacManifest, normalizedRoleId);
}
```

Route-level pre-check (defense in depth):

```js
// server/modules/projects/routes.js
{
  path: "/api/workspace/projects",
  method: "GET",
  auth: "required",
  workspacePolicy: "required",
  workspaceSurface: "admin",
  permission: "projects.read",
  ...
}
```

Auth policy plugin enforcement:

```js
// packages/auth/fastify-auth-policy/src/plugin.js
const allowed = await Promise.resolve(
  hasPermission({
    permission: meta.permission,
    permissions: request.permissions,
    request,
    actor: actorResolution.actor,
    context,
    meta
  })
);
if (!allowed) {
  throw createError(403, "Forbidden.");
}
```

Action-level authoritative check:

```js
// projects contributor
permission: ["projects.read"]
```

And pipeline evaluates it for every channel.

External calls in this stage:
- `loadRbacManifest(...)`: reads/parses JSON manifest from disk.
- `authService.authenticateRequest(request)`: resolves authenticated actor from request credentials.
- `workspaceService.resolveRequestContext(...)`: resolves workspace + membership + permission list.
- permission evaluator in action runtime: verifies required permissions against `context.permissions`.

## 6) Add a Real Form Submit: `PATCH` + Schema Usage

This is the missing piece for frontend form work: how a submitted form becomes validated server data.

### 6.1 Route declares PATCH schema

```js
// server/modules/projects/routes.js
{
  path: "/api/workspace/projects/:projectId",
  method: "PATCH",
  auth: "required",
  workspacePolicy: "required",
  workspaceSurface: "admin",
  permission: "projects.write",
  schema: {
    params: schema.params,
    body: schema.body.update,
    response: withStandardErrorResponses(
      { 200: schema.response.single },
      { includeValidation400: true }
    )
  },
  handler: controllers.projects?.update
}
```

External calls here:
- `withStandardErrorResponses(...)`: adds standard API error response contracts used by OpenAPI/docs and tests.

### 6.2 Schema defines allowed form fields

```js
// server/modules/projects/schema.js
const params = Type.Object(
  {
    projectId: Type.Integer({ minimum: 1 })
  },
  { additionalProperties: false }
);

const update = Type.Object(
  {
    name: Type.Optional(Type.String({ minLength: 1, maxLength: 160 })),
    status: Type.Optional(enumSchema(["draft", "active", "archived"])),
    owner: Type.Optional(Type.String({ maxLength: 120 })),
    notes: Type.Optional(Type.String({ maxLength: 5000 }))
  },
  {
    additionalProperties: false,
    minProperties: 1
  }
);
```

What this protects:
- `projectId` must be positive integer.
- Body must contain at least one editable field.
- Unknown fields are rejected (`additionalProperties: false`).

### 6.3 Controller maps transport payload to action input

```js
// server/modules/projects/controller.js
async function update(request, reply) {
  const response = await executeAction(actionExecutor, {
    actionId: "projects.update",
    request,
    input: {
      ...(request.body || {}),
      projectId: request.params?.projectId
    }
  });

  reply.code(200).send(response);
}
```

External calls here:
- `request.params` + `request.body`: Fastify parsed transport inputs.
- `actionExecutor.execute(...)`: enters canonical action pipeline.

### 6.4 Service applies business validation

```js
// server/modules/projects/service.js
if (Object.hasOwn(body, "name")) {
  patch.name = normalizeName(body.name, { required: true });
}

if (Object.keys(patch).length < 1) {
  throw new AppError(400, "Validation failed.", {
    details: {
      fieldErrors: {
        request: "At least one project field must be provided."
      }
    }
  });
}
```

External calls here:
- `normalizeName/normalizeStatus/...`: domain-level validation rules.
- `throw new AppError(...)`: returns stable API-friendly validation payload.

### 6.5 Validation order (important for debugging)

1. Fastify route schema validation (`params` and `body`) runs first.
2. Auth/workspace/permission pre-handler runs.
3. Action runtime pipeline runs (channel/surface/permission/input normalization).
4. Service business rules run.
5. Repository performs DB mutation.

### 6.6 What frontend form code should expect on failure

Global error handler shape:

```js
// packages/runtime/server-runtime-core/src/fastifyBootstrap.js
if (Array.isArray(error?.validation)) {
  const fieldErrors = resolveValidationFieldErrors(error);
  reply.code(400).send({
    error: "Validation failed.",
    fieldErrors,
    details: { fieldErrors }
  });
  return;
}
```

So for form UX, you can reliably map:
- `response.fieldErrors.<field>`
- fallback to `response.error`

## End-to-End Call Chain (GET projects)

For `GET /api/v1/workspace/projects`, effective path is:

1. Fastify receives request and runs `onRequest`.
2. Auth policy `preHandler` authenticates actor and resolves workspace context.
3. Route handler (`controllers.projects.list`) executes.
4. Controller calls `actionExecutor.execute({ actionId: "projects.list", ... })`.
5. Action registry resolves `projects.list` definition.
6. Pipeline enforces channel/surface/permission + input normalization.
7. Action handler calls `projectsService.list(...)`.
8. Service calls repository count/list methods.
9. Repository executes SQL via Knex/MySQL.
10. Result bubbles back to controller.
11. Controller sends HTTP 200 JSON payload.

For create/update endpoints, realtime publish happens in the action layer after mutation success (channel-neutral for `api`, `assistant_tool`, and `internal`).

---

## Endpoint build order

When growing an endpoint:

1. Start with controller contract.
2. Add repository for persistence.
3. Add service for business rules.
4. Route controller through action runtime.
5. Add realtime only for mutations.
6. Enforce permissions in both route metadata and action definition.
