# Realtime Flow Tutorial

This tutorial explains the server-side realtime flow in this app, step by step.

It follows the same style as `docs/flows/03.request.md`:

1. Show the exact functions involved
2. Follow the data as it moves
3. Explain each step in junior-friendly language

This covers both:

- inbound client websocket messages (`subscribe`, `unsubscribe`, `ping`)
- outbound server event fanout (project/chat/workspace updates)

## 1) Where realtime is mounted

`server.js` wires realtime during boot:

```js
await registerSocketIoRealtime(app, {
  authService,
  realtimeEventsService,
  workspaceService,
  path: API_REALTIME_PATH,
  redisUrl: runtimeEnv.REDIS_URL,
  requireRedisAdapter: NODE_ENV === "production",
  logger: app.log
});
```

`API_REALTIME_PATH` is versioned:

```js
const API_REALTIME_PATH = `${API_PREFIX}/realtime`; // "/api/v1/realtime"
```

## 2) App wrapper injects business policy

The app boundary function is intentionally small:

```js
// server/realtime/registerSocketIoRealtime.js
async function registerSocketIoRealtime(fastify, { authService, realtimeEventsService, workspaceService, ... }) {
  const appPolicyCallbacks = {
    isSupportedTopic,
    getTopicScope,
    isTopicAllowedForSurface,
    hasTopicPermission,
    buildSubscribeContextRequest,
    normalizeConnectionSurface,
    normalizeWorkspaceSlug
  };

  return registerRealtimeServerSocketio(fastify, {
    authService,
    realtimeEventsService,
    workspaceService,
    ...appPolicyCallbacks,
    ...
  });
}
```

Why this exists:

- Shared package owns websocket mechanics.
- App owns policy rules (topic catalog, topic scope, surface rules, permission checks).

## 3) Topic scope + surface + permission policy rules

App topic rules live in `shared/topicRegistry.js`:

```js
const REALTIME_TOPIC_REGISTRY = createTopicCatalog({
  alerts: {
    scope: "user",
    subscribeSurfaces: ["app", "admin", "console"],
    requiredAnyPermission: []
  },
  projects: {
    scope: "workspace",
    subscribeSurfaces: ["app", "admin"],
    requiredAnyPermission: ["projects.read"]
  },
  workspace_settings: {
    scope: "workspace",
    subscribeSurfaces: ["admin"],
    requiredAnyPermission: ["workspace.settings.view", "workspace.settings.update"]
  },
  chat: {
    scope: "workspace",
    subscribeSurfaces: ["app"],
    requiredAnyPermission: ["chat.read"]
  }
});
```

What this means:

- A topic declares scope (`workspace` or `user`).
- A topic can be available only on certain surfaces.
- A topic can require one of several permissions.
- Same rules are used during subscribe and during fanout.

## 4) Shared Socket.IO runtime startup

Shared runtime creates the Socket.IO server and optional Redis adapter:

```js
// registerRealtimeServerSocketio.js
const io = new SocketIoServer(fastify.server, {
  path,
  transports: ["websocket"],
  serveClient: false,
  maxHttpBufferSize: maxInboundMessageBytes,
  cors: false
});

if (normalizedRedisUrl) {
  redisClient = createRedisClient({ url: normalizedRedisUrl });
  await connectRedisClientWithTimeout(redisClient, ...);
  io.adapter(createRedisStreamsAdapter(redisClient));
}
```

Why this exists:

- Works in single-node mode (memory).
- Uses Redis Streams adapter for multi-node fanout.

## 5) Socket handshake auth middleware

Every socket connection passes through `io.use(...)`:

```js
io.use(async (socket, next) => {
  const connectionSurface = resolveConnectionSurface(socket, normalizeConnectionSurface);
  if (!connectionSurface) {
    next(createSocketMiddlewareError(REALTIME_ERROR_CODES.UNSUPPORTED_SURFACE));
    return;
  }

  const requestContext = buildSocketRequestContext(socket, path);
  const authResult = await authService.authenticateRequest(requestContext);
  if (!authResult?.authenticated || !authResult?.profile) {
    next(createSocketMiddlewareError(REALTIME_ERROR_CODES.UNAUTHORIZED));
    return;
  }

  socket.data.user = authResult.profile;
  socket.data.surface = connectionSurface;
  socket.data.requestContext = requestContext;
  socket.data.subscriptions = new Map();
  next();
});
```

Data created on connection:

- `socket.data.user`
- `socket.data.surface`
- `socket.data.requestContext`
- `socket.data.subscriptions`

## 6) Inbound message protocol

Realtime message types come from `@jskit-ai/realtime-contracts`:

```js
const REALTIME_MESSAGE_TYPES = {
  SUBSCRIBE: "subscribe",
  SUBSCRIBED: "subscribed",
  UNSUBSCRIBE: "unsubscribe",
  UNSUBSCRIBED: "unsubscribed",
  PING: "ping",
  PONG: "pong",
  EVENT: "event",
  ERROR: "error"
};
```

Inbound routing:

```js
async function handleInboundMessage(socket, messagePayload) {
  const type = String(messagePayload.type || "").trim();
  if (type === REALTIME_MESSAGE_TYPES.SUBSCRIBE) {
    await handleSubscribe(socket, messagePayload);
    return;
  }
  if (type === REALTIME_MESSAGE_TYPES.UNSUBSCRIBE) {
    await handleUnsubscribe(socket, messagePayload);
    return;
  }
  if (type === REALTIME_MESSAGE_TYPES.PING) {
    emitMessage(socket, createPongPayload(...));
    return;
  }
  emitProtocolError(socket, { code: REALTIME_ERROR_CODES.INVALID_MESSAGE });
}
```

## 7) Subscribe flow (the important part)

`handleSubscribe` does all key checks:

```js
async function handleSubscribe(socket, message) {
  const workspaceSlug = normalizeWorkspaceSlug(message?.workspaceSlug);
  const topics = normalizeTopics(message?.topics);

  if (!validateTopicsOrError(socket, requestId, topics)) return;
  if (!validateTopicSurfacesOrError(socket, requestId, topics)) return;

  const { workspaceTopics, userTopics } = partitionTopicsByScope(topics);
  if (workspaceTopics.length > 0 && !workspaceSlug) {
    ...workspace_required...;
    return;
  }

  let permissions = [];
  if (workspaceTopics.length > 0) {
    const resolvedContext = await resolveSubscribeAuthorization(socket, workspaceSlug, socket?.data?.surface || "");
    if (!resolvedContext) { ...forbidden...; return; }
    permissions = resolvedContext.permissions;
    // Join workspace topic rooms.
    ...
  }

  if (!validateTopicPermissionsOrError(socket, requestId, workspaceTopics, permissions)) return;
  if (!validateTopicPermissionsOrError(socket, requestId, userTopics, permissions)) return;

  // Join user-topic rooms (`u:<userId>:t:<topic>`) for user-scoped topics.
  for (const topic of userTopics) {
    ...
  }

  emitMessage(socket, createAckPayload({ type: "subscribed", requestId, workspaceSlug, topics }));
}
```

How workspace authorization is resolved (workspace-scoped topics only):

```js
async function resolveSubscribeAuthorization(socket, workspaceSlug, surfaceId) {
  const baseRequest = socket?.data?.requestContext || buildSocketRequestContext(socket, path);
  const subscribeRequest = buildSubscribeContextRequest(baseRequest, workspaceSlug, surfaceId);
  const context = await workspaceService.resolveRequestContext({
    user: socket?.data?.user || null,
    request: subscribeRequest,
    workspacePolicy: "required",
    workspaceSurface: surfaceId
  });
  if (!context?.workspace) return null;
  return context;
}
```

## 8) How subscribe request is shaped

App normalizer builds a workspace/surface-aware pseudo-request:

```js
// server/fastify/realtime/subscribeContext.js
function buildSubscribeContextRequest(request, workspaceSlugValue, surfaceIdValue = "app") {
  const normalizedWorkspaceSlug = normalizeWorkspaceSlug(workspaceSlugValue);
  const normalizedSurfaceId = normalizeConnectionSurface(surfaceIdValue);

  const headers = {
    ...(request?.headers || {}),
    "x-surface-id": normalizedSurfaceId,
    "x-workspace-slug": normalizedWorkspaceSlug
  };

  const surfacePrefix = String(SURFACE_REGISTRY[normalizedSurfaceId]?.prefix || "");
  const workspaceRoot = surfacePrefix ? `${surfacePrefix}/w` : "/w";
  const url = `${workspaceRoot}/${normalizedWorkspaceSlug || "none"}`;

  return { ...request, headers, url, raw: { ...(request?.raw || {}), url } };
}
```

Why this exists:

- Reuses existing workspace-context logic for websocket subscriptions.
- Keeps auth/workspace policy consistent with HTTP.

## 10) Realtime event service and envelope creation

`realtimeEventsService` creates normalized envelopes and publishes to an in-process bus:

```js
function publishProjectEvent({ operation, workspace, project, commandId, sourceClientId, actorUserId }) {
  const envelope = createEventEnvelope({
    eventType: operation === "created" ? resolvedRealtimeEventTypes.WORKSPACE_PROJECT_CREATED : ...,
    topic: resolvedRealtimeTopics.PROJECTS,
    workspace,
    entityType: "project",
    entityId: project?.id,
    commandId,
    sourceClientId,
    actorUserId,
    payload: { operation, projectId: Number(project?.id) }
  });
  realtimeEventsBus.publish(envelope);
  return envelope;
}
```

Envelope shape (conceptually):

```json
{
  "eventId": "evt_xxx",

```

## 11) Fanout path and re-authorization

Socket runtime listens to the realtime bus:

```js
const unsubscribeEventsListener = realtimeEventsService.subscribe((eventEnvelope) => {
  void fanoutEvent(eventEnvelope).catch(...);
});
```

Fanout modes:

1. User-scoped targeted fanout (`targetUserIds` + topic scope `user`) -> user-topic rooms (`u:<userId>:t:<topic>`)
2. Generic targeted fanout (`targetUserIds`) -> user rooms (`u:<userId>`)
3. Workspace-topic fanout -> room (`w:<workspaceId>:t:<topic>`)

Before sending workspace-topic events, it re-checks authorization:

```js
const receiveDecision = await canSocketReceiveEvent(socket, {
  workspaceId,
  workspaceSlug,
  topic
});

if (!receiveDecision?.allowed) {
  if (receiveDecision?.evict) {
    await evictWorkspaceSocketSubscription(socket, { workspaceId, topic });
  }
  return;
}

await emitEventMessage(socket, eventEnvelope);
```

This safety check evicts stale subscriptions when permissions change after initial subscribe.

## 12) Shutdown behavior

On server close:

```js
fastify.addHook("onClose", async () => {
  unsubscribeEventsListener();
  await new Promise((resolve) => io.close(() => resolve()));
  if (redisClient) {
    await closeRedisClientWithTimeout(redisClient, ...);
  }
});
```

Why this exists:

- Prevents listener leaks
- Closes websocket server cleanly
- Closes Redis connections safely

## 13) Common realtime failures and what to check

If subscribe fails:

1. `unsupported_surface`: verify handshake surface (`app/admin/console`)
2. `workspace_required`: at least one requested topic is workspace-scoped and `workspaceSlug` is missing/invalid
3. `unsupported_topic`: topic not in `shared/topicRegistry.js`
4. `forbidden`: topic not allowed for surface or missing permissions
5. `unauthorized`: auth cookies/session invalid

If events are not arriving:

1. Confirm mutation path actually publishes (controller/service path)
2. For user-scoped topics, confirm event includes `targetUserIds` and client subscribed to that user topic
3. For workspace-scoped topics, confirm event has correct `workspaceId`, `workspaceSlug`, and `topic`
4. Confirm client subscribed to matching topic scope (user-topic or workspace-topic room)
5. Confirm permission still valid (workspace subscriptions can be evicted after role/membership change)

## 14) Realtime debug model

Use this model while debugging:

1. **Connect**: authenticate socket + store user/surface context
2. **Subscribe**: validate message + topic scope + surface + permissions; resolve workspace context only when needed
3. **Publish**: business code emits envelope to realtime event bus
4. **Fanout**: runtime resolves fanout mode by topic scope, emits events, and evicts stale workspace subscriptions when re-auth fails
