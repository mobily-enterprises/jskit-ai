# Endpoint Delivery Playbook

Use this when adding or changing an API capability.

## Goal

Ship one capability through the canonical architecture without creating side paths.

## Steps

1. Define capability + action ID.
- Add or reuse canonical action ID in `shared/actionIds.js` and contributor ownership.

2. Add route contract.
- Route path/method + policy metadata (`auth`, `workspacePolicy`, `workspaceSurface`, `permission`).
- Add TypeBox route schema for params/query/body/response.

3. Keep controller transport-only.
- Parse `request.params`, `request.query`, `request.body`.
- Call `actionExecutor.execute(...)`.
- Set status/headers/cookies only.

4. Implement action definition in owning contributor.
- `id`, `version`, `kind`, `channels`, `surfaces`, `visibility`.
- `permission`, `idempotency`, `audit.actionName`.
- `execute(input, context, deps)` delegates to service.

5. Put business logic in service.
- Validation/normalization rules.
- Domain invariants.
- Orchestration across repositories and dependencies.

6. Put persistence in repository.
- SQL/Knex only.
- Row mapping + normalization.
- Transaction helpers where needed.

7. Add realtime only for mutators.
- Publish events after successful mutation.
- Use safe publisher wrappers (`publishSafely`).

8. Verify error shape and contracts.
- Preserve `fieldErrors` conventions.
- Do not leak provider/internal raw errors.

## Definition Of Done (Endpoint)

- Route schema and policy metadata are present.
- Controller has no business logic.
- Action pipeline path is used.
- Service and repository responsibilities are separated.
- Required tests are added/updated.
- Relevant docs are updated.
