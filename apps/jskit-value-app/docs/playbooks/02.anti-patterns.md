# Anti-Patterns Playbook

Use this as a stop-check before merging changes.

## Do Not Do These

1. Bypass action runtime from controllers.
- Bad: controller calls service/repository directly for business mutations.
- Good: controller -> `actionExecutor.execute(...)`.

2. Put business logic in controllers.
- Bad: permission/business branching and domain mutations in handler.
- Good: controller handles transport only.

3. Duplicate permission logic inconsistently.
- Bad: route allows but action denies (or vice versa) without intent.
- Good: action policy is authoritative; route metadata is defense-in-depth.

4. Mix persistence with domain logic.
- Bad: service composes raw SQL details.
- Good: repository owns SQL and row mapping.

5. Emit realtime before durable success.
- Bad: publish event before transaction commit/confirmed write.
- Good: publish after successful mutation path.

6. Introduce compatibility shims/aliases during clean-slate phases.
- Bad: hidden fallback routes/tool aliases.
- Good: one canonical path, explicit cutover.

7. Break error shape contracts.
- Bad: random error payloads for forms.
- Good: keep `error`, `details`, `fieldErrors` conventions.

8. Leak secrets or sensitive raw payloads into audit logs.
- Bad: tokens/password-like keys in audit metadata.
- Good: use sanitized metadata paths and safe audit recording.
