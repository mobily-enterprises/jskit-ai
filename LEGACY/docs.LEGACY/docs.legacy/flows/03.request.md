# Server Request Flow Tutorial

This is a practical walkthrough of what happens in this app when an HTTP request arrives.

The goal is to make the call chain clear for junior engineers by showing:

1. The exact function that first receives the request
2. Every major function that handles it after that
3. Why each step exists

We use this concrete example endpoint:

- `PATCH /api/settings/profile`

## 1) Where the request first enters

The first app-level function that touches each request is the `onRequest` hook from `registerRequestLoggingHooks`.

```js
// packages/runtime/server-runtime-core/src/fastifyBootstrap.js
app.addHook("onRequest", async (request) => {
  request[startedAtSymbol] = process.hrtime.bigint();
});
```

Why this exists:

- It records a start timestamp as soon as the request arrives.
- That timestamp is later used to compute latency.

Where it is wired:

```js
// server.js
registerRequestLoggingHooks(app, {
  requestStartedAtSymbol: REQUEST_STARTED_AT_SYMBOL,
  getPathname: safePathnameFromRequest,
  getSurface: resolveSurfaceFromPathname,
  observeRequest: ...,
  enableRequestLogs: NODE_ENV !== "test"
});
```

## 2) Auth and policy gate before handlers

After `onRequest`, Fastify runs `preHandler` hooks. The auth-policy plugin owns the main API pre-handler.

```js
// packages/auth/fastify-auth-policy/src/plugin.js
fastify.addHook("preHandler", async (request, reply) => {
  const pathname = request?.raw?.url || request?.url || "/";
  if (!String(pathname).startsWith(apiPrefix)) {
    return;
  }

  const meta = resolveAuthPolicyMeta(resolveRouteConfig(request));
  if (meta.csrfProtection && unsafeMethods.has(String(request?.method || "").toUpperCase())) {
    await enforceCsrfProtection(fastify, request, reply);
  }

  // auth, workspace resolution, permission checks...
});
```

Why this exists:

- Enforces auth, CSRF, workspace policy, and route-level permission checks.
- Rejects bad requests early, before business logic runs.

How it gets its dependencies:

```js
// server/fastify/auth.plugin.js
const registerPolicyPlugin = authPolicyPlugin(
  {
    async resolveActor(request, reply) {
      const authResult = await authService.authenticateRequest(request);
      ...
    },
    async resolveContext({ request, actor, meta }) {
      return workspaceService.resolveRequestContext(...);
    },
    hasPermission({ permission, permissions }) {
      return hasPermission(permissions, permission);
    }
  },
  ...
);
```

## 3) Route registration and route wrapper

Routes are assembled and registered through a shared route registrar.

```js
// server/fastify/registerApiRoutes.js
registerApiRouteDefinitions(fastify, {
  routes: routeList,
  resolveRequestUrl: safeRequestUrl,
  applyRoutePolicy(routeOptions, route) {
    return mergeAuthPolicy(routeOptions, {
      authPolicy: route.auth,
      workspacePolicy: route.workspacePolicy,
      workspaceSurface: route.workspaceSurface,
      permission: route.permission,
      ...
    });
  }
});
```

The actual Fastify wrapper that invokes the route handler:

```js
// packages/runtime/server-runtime-core/src/apiRouteRegistration.js
fastify.route({
  ...routeOptions,
  handler: async (request, reply) => {
    await routeHandler(request, reply, toRequestUrl(request));
  }
});
```

Why this exists:

- Keeps route registration consistent.
- Applies policy metadata in one place.
- Wraps all handlers with the same calling contract.

## 5) Controller delegates to action executor

For this endpoint, the controller does transport work and delegates business execution to actions.

```js
// packages/workspace/settings-fastify-routes/src/controller.js
async function updateProfile(request, reply) {
  const payload = request.body || {};
  const result = await executeAction(actionExecutor, {
    actionId: SETTINGS_ACTION_IDS.PROFILE_UPDATE,
    request,
    input: payload
  });

  if (result.session) {
    authService.writeSessionCookies(reply, result.session);
  }

  reply.code(200).send(result.settings);
}
```

Why this exists:

- Controller handles HTTP concerns only (body, cookies, status code, response).
- Business behavior lives in the action pipeline and domain services.

## 6) `actionExecutor` normalizes context and calls registry

```js
// server/runtime/actions/index.js
async execute({ actionId, version = null, input = {}, context = {}, deps = {} } = {}) {
  const executionContext = buildExecutionContext(context);

  return actionRegistry.execute({
    actionId,
    version,
    input,
    context: executionContext,
    deps: {
      ...(defaultDeps || {}),
      ...(deps || {})
    }
  });
}
```

The context builder gives all channels the same normalized shape:

```js
// server/runtime/actions/buildExecutionContext.js
function buildExecutionContext({ request = null, actor = null, ..., channel = "internal", ... } = {}) {
  return normalizeExecutionContext({
    actor: resolveActor({ request, actor }),
    workspace: workspace || request?.workspace || null,
    membership: membership || request?.membership || null,
    permissions: normalizedPermissions,
    surface: resolveSurface({ request, explicitSurface: surface }),
    channel,
    requestMeta: resolveRequestMeta({ request, requestMeta }),
    assistantMeta,
    timeMeta
  });
}
```

For API controllers, `channel` is set to `"api"`.

## 7) Registry resolves definition, then pipeline runs

```js
// packages/runtime/action-runtime-core/src/registry.js
async function execute({ actionId, version = null, input = {}, context = {}, deps = {} } = {}) {
  const definition = resolveActionDefinition(index, actionId, version);
  const execution = await executeActionPipeline({
    definition,
    input,
    context,
    deps,
    permissionEvaluator,
    idempotencyAdapter,
    auditAdapter,
    observabilityAdapter,
    logger
  });

  return execution.result;
}
```

Pipeline order (simplified):

```js
// packages/runtime/action-runtime-core/src/pipeline.js
ensureActionChannelAllowed(definition, normalizedContext);
ensureActionSurfaceAllowed(definition, normalizedContext);
ensureActionVisibilityAllowed(definition, normalizedContext);

const normalizedInput = await normalizeActionInput(definition, input, normalizedContext);
const permissionResolution = await authorizationEvaluator.evaluate(...);

idempotencyKey = resolveActionIdempotencyKey(definition, normalizedContext);
ensureIdempotencyKeyIfRequired(definition, normalizedContext, idempotencyKey);

const executionResult = await definition.execute(normalizedInput, normalizedContext, deps);
const normalizedResult = await normalizeActionOutput(definition, executionResult, normalizedContext);

await emitAuditEvent(normalizedAuditAdapter, ...);
```

Why this exists:

- Guarantees one execution path for policy, validation, idempotency, audit, and observability.
- Prevents controllers or tools from bypassing business protections.

Why this exists:

## 9) Domain service logic

The action calls `userSettingsService.updateProfile`.

```js
// server/modules/settings/service.js
async function updateProfile(request, user, payload) {
  const parsed = parseProfileInput(payload);
  if (Object.keys(parsed.fieldErrors).length > 0) {
    throw validationError(parsed.fieldErrors);
  }

  const updated = await authService.updateDisplayName(request, parsed.displayName);
  const profile =
    updated.profile || (await userProfilesRepository.updateDisplayNameById(user.id, parsed.displayName));
  const settings = await userSettingsRepository.ensureForUserId(profile.id);
  const chatSettings = await chatUserSettingsRepository.ensureForUserId(profile.id);
  const securityStatus = await authService.getSecurityStatus(request);

  return {
    settings: buildResponse(profile, settings, securityStatus, chatSettings),
    session: updated.session || null
  };
}
```

Why this exists:

- Domain validation and domain behavior are centralized here.
- Service composes data from auth + user settings + chat settings into one response contract.

```js
// packages/auth/auth-provider-supabase-core/src/service.js
async function syncProfileMirror(nextProfile) {
  const normalized = buildNormalizedIdentityProfile(nextProfile);
  const existing = await findProfileByIdentity(normalized);
  if (!profileNeedsUpdate(existing, normalized)) {
    return requireSynchronizedProfile(existing);
  }

  const upserted = await upsertProfileByIdentity(normalized);
  return requireSynchronizedProfile(upserted);
}
```

Why this exists:

- Source of truth for identity is the auth provider (Supabase).
- Local profile table is kept in sync for app queries and joins.

## 12) Response completion and final logging

Controller sends response:

```js
reply.code(200).send(result.settings);
```

After response is sent, the `onResponse` hook runs:

```js
// packages/runtime/server-runtime-core/src/fastifyBootstrap.js
app.addHook("onResponse", async (request, reply) => {
  const durationMs = ...;
  ...
  if (typeof observeRequest === "function") {
    observeRequest({ method, route, surface, statusCode, durationMs });
  }
  if (enableRequestLogs) {
    request.log.info(logPayload, "request.completed");
  }
});
```

Why this exists:

- Emits final request metrics/logs with status and duration.

## Request debug checklist

When you debug a request, think in this order:

1. Fastify hooks (`onRequest`, then `preHandler`)
2. Route mapping (`path + method -> controller`)
3. Controller transport mapping (`request -> action input`, `action output -> HTTP reply`)
4. Action runtime pipeline (policy/validation/permission/idempotency/audit)
5. Action handler -> domain service
6. Repositories / provider calls
7. Response + `onResponse` logs
